# 07. Тестирование И Приёмка

## Приёмка Клиентом

1. Проверить, что бот `Порядок на АЗС` присылает запрос отчёта.
2. Открыть отчёт по кнопке из сообщения.
3. Сделать фото по порядку.
4. Переснять фото до подтверждения.
5. Подтвердить фото и убедиться, что оно загружается в фоне.
6. После всех фото нажать `Отправить отчёт`.
7. Проверить, что отчёт появился у проверяющего как выполненный.
8. Открыть папку с фото по ссылке из dashboard.

## Приёмка Bitrix24-Специалистом

1. Проверить наличие СП `АЗС`, `Типы фото`, `Отчёты АЗС`.
2. Проверить типы полей и маппинг в настройках приложения.
3. Проверить стадии `new`, `in_progress`, `done`, `expired`.
4. Проверить scopes приложения.
5. Проверить, что бот зарегистрирован и пишет пользователям.
6. Проверить, что поле `Папка Диска` заполняется строковым ID.

## Приёмка Разработчиком

1. Запустить локальное окружение.
2. Проверить `/api/health`.
3. Проверить `/api/getToken` из iframe.
4. Проверить сохранение настроек.
5. Проверить ручной запуск на одной АЗС.
6. Проверить ручной запуск на нескольких АЗС.
7. Проверить автоматический запуск по ближайшему времени.
8. Проверить загрузку фото и submit.
9. Проверить timeout watcher.

## Негативные Сценарии

- Не выбранная АЗС при ручном запуске -> понятная ошибка формы.
- Невалидный `reportItemId` -> явная ошибка синхронизации CRM.
- Не заполнен `disk.folderNameTemplate` -> блокирующая ошибка настроек.
- Не заполнен `report.fields.folderId` -> блокирующая ошибка production-конфигурации.
- Истёк OAuth token -> refresh или понятная ошибка `auth_context_unavailable`.
- Пользователь без прав пытается открыть настройки -> `403`.

## Smoke Checklist Перед Демонстрацией

- Backend и frontend контейнеры запущены.
- Cloudpub URL доступен.
- Приложение открывается в Bitrix24 iframe.
- `GET /api/health` возвращает 200.
- В настройках сохранены СП, поля, стадии, роли и расписание.
- Есть хотя бы одна активная АЗС с набором обязательных фото.
- У ответственного пользователя есть доступ к приложению.
