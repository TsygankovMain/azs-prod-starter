# Деплой в Timeweb App Platform (Dockerfile-only)

Этот документ описывает деплой текущего приложения в Timeweb Cloud App Platform в режиме **одного контейнера**.

## Важное ограничение

В этой конфигурации PostgreSQL запускается внутри того же контейнера.  
При redeploy или пересоздании контейнера данные могут быть потеряны.

## Требования к RAM

Выбирайте тариф с **≥ 1 ГБ RAM**. Пиковое потребление памяти складывается из:
- базового RSS Node.js (~100–150 МБ),
- фотографий одного отчёта в памяти (multer `memoryStorage`) × 1.33 из-за base64-кодирования перед отправкой в Bitrix24.

Например, 10 фото по 3 МБ = 30 МБ × 1.33 ≈ 40 МБ на один параллельный отчёт. При пиковой нагрузке нескольких одновременных отчётов суммарное потребление может превысить 512 МБ.

## Что подготовить

1. Репозиторий с корневым `Dockerfile`.
2. Production значения переменных окружения (база: `.env.timeweb.example` или локальный файл `.env.timeweb`).
3. Домен приложения, совпадающий с настройками Bitrix24 app.

## Шаги в Timeweb

1. `App Platform` -> `Создать` -> тип `Dockerfile`.
2. Подключить репозиторий и выбрать ветку.
3. Отключить опцию автодеплоя (ручной релиз по commit/tag).
4. Указать env-переменные из `.env.timeweb`.
   Для single-container режима оставить `SERVER_HOST=http://127.0.0.1:8000`.
   Если Timeweb не дает оставить публичную переменную пустой, укажите `NUXT_PUBLIC_API_URL=https://<domain>`.
5. Не указывать `BITRIX_REST_ENDPOINT`: портал берется из OAuth-контекста Bitrix24 при установке и работе пользователя.
6. Запустить деплой.

## Поля локального приложения Bitrix24

Для текущего домена Timeweb:

- `Путь вашего обработчика`: `https://tsygankovmain-azs-prod-starter-fb7c.twc1.net`
- `Путь для первоначальной установки`: `https://tsygankovmain-azs-prod-starter-fb7c.twc1.net/install`
- `Код приложения (client_id)`: значение `CLIENT_ID` из env.
- `Ключ приложения (client_secret)`: значение `CLIENT_SECRET` из env.
- `Использует только API`: выключено.
- `Поддерживает BitrixMobile`: включено.
- `Название пункта меню Русский (ru)`: `Фото-отчет АЗС`.

Публичный URL должен быть HTTPS и без `:8080`. Порт `8080` используется только внутри контейнера Timeweb.

## Что проверять после деплоя

1. `GET https://<domain>/api/healthz` возвращает `200`.
2. UI открывается по домену.
3. `GET /api/health` остается защищенным (без JWT не должен давать `200`).
4. В Bitrix24:
   - открыть приложение в iframe;
   - выполнить `/api/getToken` сценарий;
   - сохранить настройки;
   - запустить ручной отчёт;
   - проверить загрузку фото и submit;
   - проверить bot уведомление.

## Релизный процесс (ручной)

1. Выбрать commit/tag.
2. В Timeweb запустить деплой этого commit/tag.
3. Проверить `healthz`.
4. Выполнить smoke-сценарий в Bitrix24.

## Минимальный smoke-checklist

1. Авторизация приложения проходит без ошибок.
2. Настройки сохраняются.
3. Ручной запуск отчёта работает.
4. Фото загружается, отчёт переводится в `done`.
5. Проверяющий получает уведомление.

## Подключение внешней managed-БД (Timeweb Managed PostgreSQL)

Этот раздел описывает переключение контейнера с встроенного PostgreSQL на внешний
кластер Timeweb (рекомендуется для прода — данные не теряются при redeploy).

### Предварительные условия

- Кластер Managed PostgreSQL создан в панели Timeweb Cloud.
- База данных `BD_AZS` и пользователь с нужными привилегиями созданы в кластере
  (подробные команды — в ранбуке `docs/superpowers/plans/2026-06-11-external-db-migration-runbook.md`).

### Шаги переключения

1. **Создать пользователя в кластере** через панель Timeweb или psql-клиентом.
2. **Выставить переменные в панели App Platform** (раздел «Переменные окружения»):

   ```
   EMBEDDED_POSTGRES=false
   DB_HOST=<host>.timeweb.cloud
   DB_PORT=5432
   DB_NAME=BD_AZS
   DB_USER=<db_user>
   DB_PASSWORD=<db_password>   # только через панель, не в файле
   DB_SSL=require
   ```

3. **Запустить redeploy** текущего commit/tag в App Platform.
4. **Критерий успеха** — эндпоинт `GET https://<domain>/api/healthz` возвращает `200 {"ok":true}`.
   Если возвращает `503 {"ok":false,"error":"db_unavailable"}` — проверить:
   - правильность DB_HOST / DB_PORT / DB_NAME / DB_USER в панели;
   - что пользователь имеет CONNECT-привилегию на базу;
   - что TLS-режим соответствует настройкам кластера (DB_SSL=require для Timeweb).

### Как работает флаг EMBEDDED_POSTGRES

| Значение | Поведение |
|----------|-----------|
| `true` (по умолчанию) | supervisord запускает postgres + db-init внутри контейнера; start-backend.sh ждёт маркер `/tmp/db-init.done` |
| `false` | postgres и db-init завершаются немедленно с кодом 0; start-backend.sh выполняет прямую pg_isready-проверку к DB_HOST:DB_PORT (до 60 с) |

Полный runbook миграции данных и инструкция отката:
`docs/superpowers/plans/2026-06-11-external-db-migration-runbook.md`
