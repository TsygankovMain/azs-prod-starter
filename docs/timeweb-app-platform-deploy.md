# Деплой в Timeweb App Platform (Dockerfile-only)

Этот документ описывает деплой текущего приложения в Timeweb Cloud App Platform в режиме **одного контейнера**.

## Важное ограничение

В этой конфигурации PostgreSQL запускается внутри того же контейнера.  
При redeploy или пересоздании контейнера данные могут быть потеряны.

## Что подготовить

1. Репозиторий с корневым `Dockerfile`.
2. Production значения переменных окружения (база: `.env.timeweb.example`).
3. Домен приложения, совпадающий с настройками Bitrix24 app.

## Шаги в Timeweb

1. `App Platform` -> `Создать` -> тип `Dockerfile`.
2. Подключить репозиторий и выбрать ветку.
3. Отключить опцию автодеплоя (ручной релиз по commit/tag).
4. Указать env-переменные из `.env.timeweb.example`.
   Для single-container режима оставить `SERVER_HOST=http://127.0.0.1:8000` и пустой `NUXT_PUBLIC_API_URL`.
5. Запустить деплой.

## Что проверять после деплоя

1. `GET https://<domain>/api/healthz` возвращает `200`.
2. UI открывается по домену.
3. `GET /api/health` остается защищенным (без JWT не должен давать `200`).
4. В Bitrix24:
   - открыть приложение в iframe;
   - выполнить `/api/getToken` сценарий;
   - сохранить настройки;
   - запустить ручной отчёт;
   - проверить загрузку фото и submit;
   - проверить bot уведомление.

## Релизный процесс (ручной)

1. Выбрать commit/tag.
2. В Timeweb запустить деплой этого commit/tag.
3. Проверить `healthz`.
4. Выполнить smoke-сценарий в Bitrix24.

## Минимальный smoke-checklist

1. Авторизация приложения проходит без ошибок.
2. Настройки сохраняются.
3. Ручной запуск отчёта работает.
4. Фото загружается, отчёт переводится в `done`.
5. Проверяющий получает уведомление.
