## Открытые баги (смоук прода после релиза спринта 4)

### BUG-008 · HIGH · Фотолента на проде открывается «пустой»: ни фильтров, ни «Нет фото», только кнопка «Показать ещё»
- **Где:** прод b24-xc36ra, десктоп, рабочий стол «Фотолента АЗС», вкладка Фотолента, первый заход после деплоя `1f76243`.
- **Симптом (скриншот продакта 2026-06-11):** шапка и вкладки рендерятся, но левая панель фильтров отсутствует ПОЛНОСТЬЮ, пустого состояния «Нет фото за выбранный период» нет, ошибки нет — висит одинокая кнопка «Показать ещё». Консоль: 0 ошибок (только известный шум Violations от b24ui).
- **Анализ кода (исключено):** PhotoFilters рендерится безусловно (photos.client.vue:614, корень компонента без v-if); «Показать ещё» требует `nextCursor` (:662), который бэк не отдаёт при пустой выдаче (limit+1-паттерн); пустое состояние перекрывает сетку при `!isLoading && items.length===0` (:639). Видимая комбинация недостижима при согласованном бандле и штатном ответе API.
- **Гипотеза №1 (ведущая):** смешанный/устаревший кэш чанков SPA после редеплоя в давно открытой вкладке Битрикса (часть чанков старые, часть новые → ленивые компоненты тихо не поднялись). Лечится жёстким обновлением; если подтвердится — системный фикс: cache-headers/версионирование ассетов в nginx-конфиге контейнера + «требуется обновление» баннер.
- **Гипотеза №2:** дефолт «Сегодня» честно пуст (фото загружаются только при сдаче отчётов), а видимая аномалия — следствие №1; после жёсткого обновления ожидаем корректное «Нет фото за выбранный период» + фильтры, и тогда останется только UX-вопрос «первый экран пустой по умолчанию» (кандидат: дефолт «Неделя» или авто-расширение периода при пустом «Сегодня»).
- **РАЗВЯЗКА (2026-06-11, после жёсткого обновления у продакта):** кэш-гипотеза опровергнута; Network показал — `feed` возвращает ~6 КБ данных (фото есть!), а сетка/фильтры всё равно пустые. Корень подтверждён реестром сборки `.nuxt/components.d.ts`: Nuxt по умолчанию регистрирует компоненты из подпапок с префиксом пути (`components/reports/R1Summary.vue` → `ReportsR1Summary`), а ВЕСЬ код приложения зовёт их по имени файла (`<R1Summary>`, `<PhotoFilters>`). В прод-сборке Vue молча рендерит неизвестный тег в пустоту — ни ошибки в консоли, ни падения lint/build.
- **Реальный масштаб (одним корнем):** фотолента целиком (5 компонентов), страница «Отчёты АЗС» R1–R5 (включая внутренние SvgRing/ReportNav — была сломана на проде С МОМЕНТА ЕЁ РЕЛИЗА, никто не открывал), мокапы в справке (4 компонента). Сломан весь слой вложенных компонентов.
- **Фикс (как блокер прода, вне фикс-спринта):** `nuxt.config.ts` → `components: [{ path: '~/components', pathPrefix: false }]` — регистрация по имени файла, как написан весь код. Проверено: все 26 имён файлов уникальны (коллизий нет), префиксными именами никто не пользовался, реестр после `nuxt prepare` беспрефиксный, build зелёный.
- **Урок в копилку процессов:** «lint+build зелёные» НЕ ловят нерезолвящиеся auto-import-компоненты; в DoD спринтов добавить визуальный прогон новых страниц (dev-сервер или скриншот-смоук) до релиза.
- **Статус:** ✅ ЗАКРЫТ — хотфикс задеплоен 15:08 (`05a4d84`), продакт подтвердил скриншотами: фильтры, категории, сетка с тайлами и флажками живые. Добивки смоука — отдельными багами ниже.

### BUG-013 · HIGH · Превью фото в ленте не загружаются (тайлы тёмные, без картинок)
- **Где:** прод после хотфикса BUG-008, Фотолента, АЗС «Тестовая» №174, свежие фото 14:47–14:48 (отчёт «Сдан», папка 106076).
- **Симптом (подтверждено продактом):** тайлы рендерятся с подписями и флажками, но картинки-превью не прогружаются — однотонные тёмные плитки.
- **Трасса по коду:** грид лениво зовёт `apiStore.getPhotoPreviewObjectUrl` → `GET /api/reports/photos/:reportId/:photoCode/preview` (api.ts:537) → analyticsRoutes.js:81 → `diskApi.downloadFileContent(disk_object_id)` под admin-контекстом; ошибки бэка: 501 `preview_not_supported` / 502 `preview_failed` (:100,:115). Коллизий монтирования с photoFeedRouter нет (проверено живым зондом в Opus-ревью A2). Admin-контекст жив (feed и категории грузятся им же).
- **Кандидаты:** (а) 502 — `downloadFileContent` падает на этих disk_object_id (правá/скоуп диска, размер, тип файла); (б) фронт глотает ошибку молча: previewErrors помечает тайл и НЕ показывает состояние «не удалось» и не ретраит (PhotoFeedGrid.vue:35,41) — отдельная UX-добивка в любом случае; (в) маловероятно — 401/404 по маршруту.
- **Недостающее свидетельство (запрошено у продакта):** Network → фильтр «preview» → статус-код и тело ответа одного запроса.
- **Фикс-скоуп (когда подтвердим слой):** починить причину на бэке + на фронте видимое состояние ошибки превью с ретраем по тапу; тест на 502-путь.
- **Дополнение (скриншот «Фото-витрина», 15:14):** превью так же не грузятся и в витрине R5 — общесистемно (один прокси-эндпоинт), не специфично для фотоленты.

### BUG-014 · HIGH · «Карточка АЗС» (R4) показывает 0/0 и «нет истории», хотя сдачи есть
- **Где:** прод, Отчёты АЗС → вкладка «Карточка», АЗС «Тестовая», за 30 дней.
- **Симптом (скриншот продакта 15:14):** «% вовремя 0%», «Сдач всего 0/0», «История прохождений: нет истории за 30 дней». При этом ФАКТ: сегодня отчёт Тестовой СДАН в 14:48 (та же страница, вкладка «Фото-витрина», видит сдачу и 4 фото), вчера — провален. Отчётность противоречит сама себе.
- **Трасса по коду:** R4Card строится НЕ отдельной аналитикой, а списком `GET /api/reports?dateFrom&dateTo&azsId&limit=50` (R4Card.vue:60) с клиентским подсчётом KPI. Витрина же идёт через `/analytics/day-photos` без фильтра azsId — поэтому видит. Контракт параметров сходится (normalizeAzsIds корректно ест скаляр, reportsRoutes.js:68), фильтры стора покрыты тестами → подозрение на ДАННЫЕ или конкретный запрос.
- **Кандидаты:** (а) значение в выпадашке карточки (id элемента CRM из /azs-options) ≠ `dispatch_log.azs_id`, который пишет рассылка (вкл. ручную) — рассинхрон id-пространств; (б) дата-фильтр normalizeDateFilter отсекает всё (формат ISODate vs ожидание); (в) баг выборки именно со связкой azsIds+даты.
- **Недостающее свидетельство (запрошено у продакта):** Network → фильтр `reports?` → URL запроса карточки (какой azsId ушёл) и ответ (items: сколько).
- **Фикс-скоуп:** выровнять id/фильтр + регресс-тест «карточка видит то же, что витрина»; заодно решить продуктово: KPI карточки честнее строить на бэке (один источник правды), а не клиентским пересчётом 50 строк.

### BUG-009 · HIGH · Бот не отправляет сообщение при запросе отчёта; рассылка 12:00 — все АЗС в «Ошибке»
- **Где:** прод, 2026-06-11. Скриншот тех-таблицы: отчёты 528–541 (слот `2026-06-11:1200`) — ВСЕ со статусом «Ошибка»; ручная рассылка 543 (`manual:…:1146`, срок 13:20Z) висит «Запланирован» спустя ~2.5 часа после срока. При этом 542 (manual 11:45) — «Сдан», т.е. утром бот работал.
- **Факты из кода:** причина каждой «Ошибки» УЖЕ сохранена в `dispatch_log.error_text` (dispatchLogStore.js:34,77) — но нигде не показывается (см. BUG-011). При пустом auth-контексте планировщик молча пишет warn `missing auth/webhook context` и пропускает тик (dispatchScheduler.js:240,325); слот, обработанный «вхолостую», помечается processed и не повторяется (:458).
- **Хронология-гипотеза:** 12:00-слот и 13:20-ручная пришлись на период ДО деплоя спринта 4 (14:42) — старый код с файловым auth-контекстом, потерянным предыдущим редеплоем → массовый сбой отправки; после 14:42 новый код с composite-стором должен сам себя вылечить (контекст засеян входом админа), но уже «сгоревшие» слоты не ретраятся.
- **Что чинить (фикс-спринт):** (1) показать error_text по-человечески (= BUG-011); (2) механизм добивания: просроченные «Запланирован»-слоты после восстановления контекста либо ретраятся, либо честно переводятся в «Ошибка: не было доступа к Битриксу в момент отправки»; (3) ручная рассылка при недоступном контексте должна сразу отвечать понятной ошибкой, а не зависать в «Запланирован».
- **Проверка-развилка (запрошена у продакта):** повторить ручную рассылку на тестовую АЗС СЕЙЧАС — если сообщение приходит, живой код здоров и баг сужается до «наследие + отсутствие ретрая/расшифровки»; если молчит — нужен лог приложения из панели Timeweb за минуту теста.
- **РЕЗУЛЬТАТ РАЗВИЛКИ (15:2x, после деплоя спринта 4 и хотфикса):** бот СНОВА не прислал уведомление при ручном запросе — баг ЖИВОЙ на новом коде, не только наследие. Подозрение смещается с auth-контекста (он жив: лента/категории/витрина работают админ-контекстом) на сам бот: BOT_ID невалиден/бот разрегистрирован на портале, либо ошибка в notify-цепочке. Причина лежит в `dispatch_log.error_text` новой строки и в логах приложения.
- **Предложение продакта (принято в фикс-скоуп):** перерегистрация бота. Механика уже есть в коде: `botRegistryService.registerBot/ensureBot` (imbot.v2.Bot.register, вызывается при /api/install; runtime-перерегистрация — botRegistryService.js:126). Деплой для этого НЕ нужен. Фикс-скоуп: (а) админ-ручка `POST /api/admin/bot/reregister` (по образцу refresh-avatar, server.js:391) + кнопка в настройках; (б) self-heal: при ошибке BOT_ID в отправке — автоматический ensureBot и один повтор; (в) понятный error_text.
- **Недостающее свидетельство:** статус и (из БД/логов) error_text НОВОЙ строки ручной рассылки; лог приложения из панели Timeweb за минуту нажатия.

### BUG-010 · PERF · Выборки из Битрикс24 не используют режим huge-data — отсюда долгие загрузки
- **Источник:** продакт + официальный гайд apidocs.bitrix24.ru/settings/performance/huge-data.html.
- **Рекомендации дока:** `start=-1` (отключает COUNT — основной тормоз), сортировка `ORDER BY ID ASC`, курсор фильтром `>ID` от последнего полученного, не запрашивать count без необходимости, стоп по пустой/неполной странице. Замер дока: 2.39 млн записей — 49.9 с → 0.097 с.
- **Наши места (аудит для фикс-спринта):** `bitrixRestClient.listCrmItems` (bitrixRestClient.js:549-576 — классический `start=0→next`, под замену в первую очередь; на нём CRM-синк и справочники), все потребители: crmSync (полные обходы smart-process), категории фотоленты, кандидаты рассылки, resync. Критерий приёмки: полный CRM-синк и первая загрузка справочников ускоряются кратно; поведение пагинации покрыто тестами с фейковым клиентом.

### BUG-011 · MEDIUM · «Ошибка» в тех-таблице и ленте проверяющего без человеческой расшифровки
- **Где:** прод, рабочий стол проверяющего, «Показать техническую информацию по отчётам» — красный бейдж «Ошибка» без какого-либо объяснения; в ленте событий тип ⚠ failed — то же.
- **Факты:** `dispatch_log.error_text` хранит причину каждого сбоя, бэк его НЕ отдаёт в выдачу отчётов/событий, фронт не показывает. Словарь useErrorText (BUG-007) уже есть — расширить серверными причинами (типовые: нет auth-контекста, Bitrix 5xx/таймаут, бот не установлен, оператор не найден).
- **Что чинить:** отдать error_text (+нормализованный errorCode) в API списка отчётов/событий → бейдж «Ошибка» с тултипом/раскрытием: человеческий текст + «Подробности» с сырой причиной (паттерн BUG-007). Покрыть тестом выдачу error_text.
