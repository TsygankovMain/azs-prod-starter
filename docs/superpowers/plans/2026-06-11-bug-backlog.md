# Бэклог багов и долгов

Правило (продакт, 2026-06-11): баги не чиним по одному — фиксируем здесь, набираем объём на спринт, исправляем пакетом. Исключение — блокеры прода.

## ✅ BUG-023 · ЗАКРЫТ (2026-06-12) · Превью не доходили до бэкенда — сломанный URL на фронте

**Корень (Network-лог + эмпирически):** `getPhotoPreviewObjectUrl` строила URL сырой склейкой `` `${apiUrl}/api/...` `` (api.ts:551), а `apiUrl="https:/"` (из `config.public.apiUrl="https://"` без хоста; SPA `ssr:false`, NUXT_PUBLIC_API_URL кривой) → `https://api/reports/...` → хост `api` → `net::ERR_NAME_NOT_RESOLVED`. Запрос вообще не долетал до сервера; прочие 30+ вызовов через `$api`/ofetch `withBase` тот же apiUrl резолвят как относительный путь → работали (фид грузился, превью нет).
**Фикс:** превью переведено на `$api(..., responseType:'blob')` — устойчиво к кривому apiUrl, +401-retry (`8c44915`→master `0ebd952`, задеплоено). **Смоук 2026-06-12 ~13:16: фото-витрина грузит тайлы (1116 · 28 фото) — превью работают end-to-end.**
**Урок:** «не грузятся картинки» → СНАЧАЛА Network (реальный URL/код), потом auth. Backend webhook-фикс BUG-020 был не в том слое для этого симптома.
**Долг (defense-in-depth):** поправить `NUXT_PUBLIC_API_URL` в Timeweb на пусто/полный origin — сейчас резолвится в `https://`.

## ✅ BUG-024 · ЗАКРЫТ (2026-06-15 спринт 7) · Отчёты рождаются уже просроченными — дедлайн в прошлом на момент отправки

**Симптом (скрины прода 17:12-17:14):** сообщение бота отправлено в **17:14**, а в нём «Сдать до **14:58**» — дедлайн в прошлом уже при отправке. Лента событий: «Создан отчёт для Тестовая» 17:12 → «Тестовая — отчёт не сдан вовремя» **17:13** (через минуту). timeoutWatcher мгновенно переводит карточку в ПРОСРОЧКУ — раньше, чем сотрудник успевает открыть сообщение.
**Корень:** `deadlineAt` считался от **планового** scheduled_at (слот 12:00 + джиттер до ±240 мин) + 40 мин на сдачу, а не от **фактического** времени отправки `notifyDispatch`. При большом джиттере / догоне зависших слотов / ручном «Запросить сейчас» дедлайн оказывался в прошлом.
**Фикс (спринт 7):** `deadlineAt = max(scheduled_at + timeout, current_time + timeout)` вместо простого `scheduled_at + timeout`. Теперь дедлайн всегда не раньше времени фактической отправки (коммиты `dddd11e`, `327825a`). Смоук: в сообщении «Сдать до» время ≥ отправки + timeout; статус не переходит в ПРОСРОЧКУ раньше timeout.

### BUG-019 · HIGH · Кнопки бота ведут диплинком в приложение, а диплинки в Б24 не работают
- **Где:** прод, личные сообщения бота «Порядок на АЗС». Кнопки ТЕПЕРЬ приходят (формат keyboard починили в v3.0), но ведут не туда.
- **Жалоба продакта (дословно):** «быстрая ссылка на приложение не будет работать в битрикс24 так как нет диплинков, мы отражали это в документации. так же выпадашка ведет на приложение, а должны быть кнопки бота.»
- **Подтверждено кодом + докой Б24:**
  - Кнопка «Открыть приложение» = `LINK` на `/marketplace/view/<APP_CODE>/?params[reportId]=N&params[path]=/admin/N` (reportLinks.js:20-31, dispatchService.js:235-252). `LINK` в keyboard Б24 = открыть URL в браузере/WebView; на мобильном клиенте уходит во ВНЕШНИЙ браузер, а не в слайдер приложения. Нативного механизма «открыть REST_APP_URI-плейсмент на экране N по клику из чата» в Б24 НЕТ (типы кнопок: LINK / COMMAND / ACTION / APP_ID(устар.)).
  - **Это было известно и осознанно отключено:** изначально кнопки стояли за флагом `ENABLE_REPORT_DEEP_LINK` (default false, «до апдейта мобилки» — specs/2026-05-31-azs-v2-design.md:51; project-log.md:328,345,371 фиксируют «bot button opened wrong page», «mobile could open external browser»). В фикс-спринте 5 (BUG-001) флаг УБРАН, гейт заменён на `BITRIX_APP_CODE` → кнопки включились безусловно. **Регресс: мы выкатили поведение, которое автор намеренно держал выключенным.**
- **«Выпадашка ведёт на приложение, а должны быть кнопки бота»** — ⚠️ ТРЕБУЕТ УТОЧНЕНИЯ продакта (скриншот). Варианты: (C1) контекстное меню/«троеточие» сообщения бота со стандартным пунктом открытия ссылки; (C2) поведение LINK-кнопки на мобильном — popup/sheet «открыть ссылку» вместо приложения; (C3) что-то в самой ленте. Не интерпретируем наугад.
- **Что нужно решить продуктово (для фикс-спринта):** какой механизм взамен — (а) `COMMAND`-кнопки бота (нажатие → событие `ONIMBOTV2COMMANDADD` → бот отвечает в чате; приложение не открывает, но не ведёт в битый браузер); (б) оставить только текст без кнопок-ссылок; (в) проверить на смоуке реального мобильного клиента 2026 — возможно, апдейт мобилки уже научился открывать `marketplace/view` нативно (тогда вернуть как есть). Решение — за продактом.
- **Недостающее свидетельство:** скриншот/видео «выпадашки»; поведение кнопки на реальном телефоне (внешний браузер? слайдер? ошибка?).

### ✅ BUG-020 · ЗАКРЫТ (2026-06-12) · Превью фото не грузятся
- **Итог:** реальный корень оказался фронтовым (BUG-023 — битый URL `https://api/...`, запрос не доходил до бэка). После фронт-фикса превью грузятся end-to-end (смоук 2026-06-12 ~13:16, фото-витрина 1116 · 28 фото). Значит и авторизация скачивания (disk.file.get) сейчас отрабатывает — через вебхук (BUG-020-фикс `740eace`) или admin-контекст. Авто-диагностика ниже оставлена как историческая; архитектурный долг «серверные thumbnails вместо base64» — в «Отложенном».
- **Где:** прод, фотолента и фото-витрина; ВСЕ тайлы «Не удалось / Нажмите для повтора» (error-UI v3.0 работает, но картинки не приходят). Не грузятся даже СЕГОДНЯШНИЕ фото (14:47-14:48) → корень НЕ в disk_object_id, а в авторизации скачивания.
- **Сценарий рендера (для продакта, человеческим языком):** оператор грузит фото → файл кладётся на Диск Б24, в нашу БД пишется `disk_object_id` (report_photo). Тайл ленты при попадании в видимую зону зовёт `GET /api/reports/photos/:id/:code/preview` → бэк находит `disk_object_id` → берёт admin-контекст → `disk.file.get` → получает временную ссылку DOWNLOAD_URL → качает файл → отдаёт картинку. Если на любом шаге токен невалиден — тайл показывает «Не удалось».
- **Корень (по коду, ранжировано):**
  1. **Admin-контекст протух / отсутствует в БД** (`getAdminContext`→`getLastAdminContext`); после редеплоя/долгого простоя authId истёк, refresh нужен.
  2. **CLIENT_ID/SECRET в панели не починены** → `refreshAccessToken` падает `invalid_client` → 401-retry бесполезен (bitrixRestClient.js:104-106). **СВЯЗКА: пока env не починен И admin-контекст не пересеян свежим входом — превью не оживут.**
  - downloadFileContent уже умеет 401→refresh→retry (v3.0), но retry упирается в тот же сломанный refresh.
- **Диагностика за 2 минуты (нужно с прода):** DevTools → Network → запрос `preview` → тело 502 (поле `message`: `invalid_client` / `expired_token` / `Disk download failed HTTP 401`); лог прода `grep oauth_client_invalid|preview_failed`; `SELECT is_admin,last_admin_at FROM auth_context WHERE is_admin=true`.
- **Вероятный путь лечения:** (1) починить CLIENT_ID/SECRET в панели; (2) зайти в приложение порталным админом → `/api/install`/`getToken` пересеет свежий admin-контекст в БД → tokenRefreshScheduler держит его живым. Если после этого свежие фото грузятся — баг закрыт средой; если нет — копать права admin на файлы Диска/формат DOWNLOAD_URL.
- **Архитектурная заметка (на подумать):** превью качается полным файлом в base64 через наш бэк — тяжело и медленно даже при живом токене. Кандидат на серверные thumbnails / отдачу по подписанному URL (отложенное).

### BUG-021 · HIGH · Отчёты формируются очень долго — N+1 запросов названий АЗС (huge-data применён не туда)
- **Где:** прод, «Отчёты АЗС» — вкладки R2 Рейтинг, R4 Карточка, R5 Фото-витрина грузятся секундами/десятками секунд.
- **Корень (доказан кодом):** `createAzsTitleResolver` (reportsRoutes.js:109-161) резолвит название КАЖДОЙ АЗС отдельным `crm.item.get` — N+1. R2 (analyticsRoutes.js:29-33), R4 (reportsRoutes.js:789-796), R5 (day-photos/feed) декорируют строки через `Promise.all(rows.map(resolve))` → при холодном кэше десятки последовательных HTTP к Б24 (~50 АЗС × 200-500мс = 10-25с; +rate-limit/503).
- **Почему huge-data (BUG-010) не помог:** `start=-1`-режим применён только к `listCrmItems` (выпадашка /azs-options, кандидаты рассылки). Резолвер названий на горячем пути отчётов использует `getCrmItem` поштучно — мимо оптимизации. **Это ровно то, что продакт видит в своём другом приложении: там названия тянут одним списком, у нас — по одному.**
- **Рецепт (фикс-спринт):** `batchResolveAzsTitles(azsIds[])` — один `crm.item.list({entityTypeId, select:['id','title'], filter:{}, start:-1})` → Map id→title; подключить в R2/R4/R5/feed вместо поштучного resolve (собрать уникальные azsId страницы → один вызов → маппинг). `createAzsTitleResolver` оставить для одиночных мест (карточка/submit/upload). Файлы: reportsRoutes.js:109-161,780-810, analyticsRoutes.js:27-33,67-73, photoFeedRoutes.js:121-149. Ожидаемо: 15с → ~0.5с.

## ⚠️ BUG-022 · BLOCKER · `wrong_client` — в env стоят ключи ЛОКАЛЬНОГО приложения, а прод установлен из МАРКЕТПЛЕЙСА

**Это, вероятно, мастер-корень BUG-009 (рассылка засыпает) и BUG-020 (превью), а не отдельные баги.**

- **Факт от продакта (2026-06-11):** приложение на проде авторизуется как ТИРАЖНОЕ (из Маркетплейса) → client_id/client_secret должны быть от маркетплейс-приложения.
- **Что в env сейчас:** `CLIENT_ID=local.6a0cbb25…` (формат ЛОКАЛЬНОГО приложения). Несоответствие.
- **Документация Б24 (проверено по docs, не по памяти):**
  - [oauth/index.html](https://apidocs.bitrix24.ru/api-reference/oauth/index.html) и [oauth/auth-renewal](https://apidocs.bitrix24.ru/api-reference/oauth/auto-renewal.html): «client_id / client_secret — код/секрет приложения, получаемый **в партнёрском кабинете** при регистрации [тиражного], **либо на портале в случае локального**».
  - Refresh уходит на `https://oauth.bitrix24.tech/oauth/token/` (центральный сервер), не на портал. Все операции с client_secret — только с этим сервером.
  - Тиражный client_id имеет префикс `app.`, локальный — `local.`. refresh_token живёт **180 дней** (наша прежняя оценка «30 дней» неверна — возрастной протух токена тут НИ ПРИ ЧЁМ; проблема — несовпадение client_id/secret).
- **Механизм отказа:** фоновые операции без пользователя (dispatchScheduler, timeoutWatcher, скачивание превью с Диска) обновляют access_token server-side через CLIENT_ID/CLIENT_SECRET. Если они от `local.`-приложения, а установлено `app.`-приложение → `invalid_client`/`wrong_client` на каждом refresh → шторм в логах (2399×) → рассылка жива только «пока админ активен» (его свежий токен прилетает через getToken/BX24 SDK), а превью (нужен свежий токен для disk.file.get) не оживают.
- **РЕШАЮЩАЯ ПРОВЕРКА (для Cowork):** какой client_id у приложения, РЕАЛЬНО установленного на b24-xc36ra.bitrix24.ru? Если `app.XXXX` — это маркетплейс, и в env Timeweb надо вписать `app.XXXX` + его секрет из КАБИНЕТА РАЗРАБОТЧИКА/ПАРТНЁРА (где приложение опубликовано), а НЕ со страницы приложения на портале. Если `local.6a0cbb25…` — тогда локальное, и неверен только секрет (берётся с портала).
- **Мой прежний совет «перевбить со страницы приложения на портале» — НЕВЕРЕН для тиражного приложения. Источник правды — партнёрский кабинет.**
- **Критерий закрытия:** после исправления env + рестарта через ~1 час простоя в логах нет `wrong_client`; рассылка работает без входа админа; превью грузятся сами. Тогда BUG-009 и BUG-020 закрываются этим же действием.

### Заметка по коду (для фикс-спринта, связано с BUG-022)
- Проверить, как `refreshAccessToken` (bitrixRestClient.js) формирует URL OAuth-сервера: должен быть `oauth.bitrix24.tech` (или `server_endpoint`/`server_domain` из install-payload), НЕ портал. Сверить с тем, что приходит в `/install` (`server_endpoint`, `member_id`) — возможно, надо хранить и использовать server_endpoint портала, а не хардкод.
- Сделать стартовую самопроверку: при `invalid_client` на refresh — один раз залогировать ЧЕЛОВЕЧЕСКИ «CLIENT_ID/SECRET не соответствуют установленному приложению (local vs app?)», а не тонуть в throttled-шторме.

## ⚠️ Действие продакта вне кода (обязательное, БЕЗ деплоя)

**CLIENT_ID / CLIENT_SECRET в панели Timeweb** — перевбить оба значения со страницы приложения на портале (Разработчикам → наше приложение), перезапустить. Критерий: через ~1 час простоя в логах нет `wrong_client`. Без этого фоновая рассылка живёт только «пока админ заходит в приложение» (BUG-009 п.2 / BUG-017-шторм).
