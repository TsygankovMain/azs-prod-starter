# Бэклог багов и долгов

Правило (продакт, 2026-06-11): баги не чиним по одному — фиксируем здесь, набираем объём на спринт, исправляем пакетом. Исключение — блокеры прода.

## Открытые баги (смоук прода после релиза спринта 4)

### BUG-008 · HIGH · Фотолента на проде открывается «пустой»: ни фильтров, ни «Нет фото», только кнопка «Показать ещё»
- **Где:** прод b24-xc36ra, десктоп, рабочий стол «Фотолента АЗС», вкладка Фотолента, первый заход после деплоя `1f76243`.
- **Симптом (скриншот продакта 2026-06-11):** шапка и вкладки рендерятся, но левая панель фильтров отсутствует ПОЛНОСТЬЮ, пустого состояния «Нет фото за выбранный период» нет, ошибки нет — висит одинокая кнопка «Показать ещё». Консоль: 0 ошибок (только известный шум Violations от b24ui).
- **Анализ кода (исключено):** PhotoFilters рендерится безусловно (photos.client.vue:614, корень компонента без v-if); «Показать ещё» требует `nextCursor` (:662), который бэк не отдаёт при пустой выдаче (limit+1-паттерн); пустое состояние перекрывает сетку при `!isLoading && items.length===0` (:639). Видимая комбинация недостижима при согласованном бандле и штатном ответе API.
- **Гипотеза №1 (ведущая):** смешанный/устаревший кэш чанков SPA после редеплоя в давно открытой вкладке Битрикса (часть чанков старые, часть новые → ленивые компоненты тихо не поднялись). Лечится жёстким обновлением; если подтвердится — системный фикс: cache-headers/версионирование ассетов в nginx-конфиге контейнера + «требуется обновление» баннер.
- **Гипотеза №2:** дефолт «Сегодня» честно пуст (фото загружаются только при сдаче отчётов), а видимая аномалия — следствие №1; после жёсткого обновления ожидаем корректное «Нет фото за выбранный период» + фильтры, и тогда останется только UX-вопрос «первый экран пустой по умолчанию» (кандидат: дефолт «Неделя» или авто-расширение периода при пустом «Сегодня»).
- **Недостающие свидетельства (запрошены у продакта):** (а) результат Cmd+Shift+R; (б) Network: статус и тело `GET /api/reports/photos/feed` после «Обновить»; (в) виден ли запрос `_nuxt/*.js` со статусом 404.
- **Статус:** зафиксирован, ждёт фикс-спринта (или мгновенного подтверждения гипотезы №1 продактом).

### BUG-009 · HIGH · Бот не отправляет сообщение при запросе отчёта; рассылка 12:00 — все АЗС в «Ошибке»
- **Где:** прод, 2026-06-11. Скриншот тех-таблицы: отчёты 528–541 (слот `2026-06-11:1200`) — ВСЕ со статусом «Ошибка»; ручная рассылка 543 (`manual:…:1146`, срок 13:20Z) висит «Запланирован» спустя ~2.5 часа после срока. При этом 542 (manual 11:45) — «Сдан», т.е. утром бот работал.
- **Факты из кода:** причина каждой «Ошибки» УЖЕ сохранена в `dispatch_log.error_text` (dispatchLogStore.js:34,77) — но нигде не показывается (см. BUG-011). При пустом auth-контексте планировщик молча пишет warn `missing auth/webhook context` и пропускает тик (dispatchScheduler.js:240,325); слот, обработанный «вхолостую», помечается processed и не повторяется (:458).
- **Хронология-гипотеза:** 12:00-слот и 13:20-ручная пришлись на период ДО деплоя спринта 4 (14:42) — старый код с файловым auth-контекстом, потерянным предыдущим редеплоем → массовый сбой отправки; после 14:42 новый код с composite-стором должен сам себя вылечить (контекст засеян входом админа), но уже «сгоревшие» слоты не ретраятся.
- **Что чинить (фикс-спринт):** (1) показать error_text по-человечески (= BUG-011); (2) механизм добивания: просроченные «Запланирован»-слоты после восстановления контекста либо ретраятся, либо честно переводятся в «Ошибка: не было доступа к Битриксу в момент отправки»; (3) ручная рассылка при недоступном контексте должна сразу отвечать понятной ошибкой, а не зависать в «Запланирован».
- **Проверка-развилка (запрошена у продакта):** повторить ручную рассылку на тестовую АЗС СЕЙЧАС — если сообщение приходит, живой код здоров и баг сужается до «наследие + отсутствие ретрая/расшифровки»; если молчит — нужен лог приложения из панели Timeweb за минуту теста.

### BUG-010 · PERF · Выборки из Битрикс24 не используют режим huge-data — отсюда долгие загрузки
- **Источник:** продакт + официальный гайд apidocs.bitrix24.ru/settings/performance/huge-data.html.
- **Рекомендации дока:** `start=-1` (отключает COUNT — основной тормоз), сортировка `ORDER BY ID ASC`, курсор фильтром `>ID` от последнего полученного, не запрашивать count без необходимости, стоп по пустой/неполной странице. Замер дока: 2.39 млн записей — 49.9 с → 0.097 с.
- **Наши места (аудит для фикс-спринта):** `bitrixRestClient.listCrmItems` (bitrixRestClient.js:549-576 — классический `start=0→next`, под замену в первую очередь; на нём CRM-синк и справочники), все потребители: crmSync (полные обходы smart-process), категории фотоленты, кандидаты рассылки, resync. Критерий приёмки: полный CRM-синк и первая загрузка справочников ускоряются кратно; поведение пагинации покрыто тестами с фейковым клиентом.

### BUG-011 · MEDIUM · «Ошибка» в тех-таблице и ленте проверяющего без человеческой расшифровки
- **Где:** прод, рабочий стол проверяющего, «Показать техническую информацию по отчётам» — красный бейдж «Ошибка» без какого-либо объяснения; в ленте событий тип ⚠ failed — то же.
- **Факты:** `dispatch_log.error_text` хранит причину каждого сбоя, бэк его НЕ отдаёт в выдачу отчётов/событий, фронт не показывает. Словарь useErrorText (BUG-007) уже есть — расширить серверными причинами (типовые: нет auth-контекста, Bitrix 5xx/таймаут, бот не установлен, оператор не найден).
- **Что чинить:** отдать error_text (+нормализованный errorCode) в API списка отчётов/событий → бейдж «Ошибка» с тултипом/раскрытием: человеческий текст + «Подробности» с сырой причиной (паттерн BUG-007). Покрыть тестом выдачу error_text.
