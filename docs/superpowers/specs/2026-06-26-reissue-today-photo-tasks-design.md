# Дизайн: «Перевыпустить задания на сегодня» (смена расписания среди дня)

**Дата:** 2026-06-26
**Источник:** продакт (чат + скрин). «Мне нужно стереть задания на сдачу фотоотчёта на сегодня, так как у нас поменялось расписание; такого функционала нет».
**Статус:** дизайн на согласовании.

## Проблема

Расписание рассылки может поменяться среди дня. Сегодняшние задания на фотоотчёт уже частично созданы под старое расписание: часть разослана сотрудникам АЗС с дедлайном по старому времени, часть ещё в плане. Сейчас отменить/снести их нечем — единственная очистка (перегенерация плана) удаляет только ещё-не-разосланные плановые слоты (`status='planned'`), а уже разосланные задания, их карточки в CRM и отправленные сообщения не трогает. Уведомления «задание отменено» нет.

## Цель

Одно админское действие **«Перевыпустить задания на сегодня»**, которое:
1. снимает все **несданные** сегодняшние задания по **всем АЗС** (сданные с фото — не трогает);
2. уведомляет затронутых сотрудников АЗС сообщением бота об отмене;
3. **пересоздаёт** задания на сегодня по текущему расписанию из настроек (новые задания уходят сами на новых временах через штатный планировщик).

### Решения продакта (зафиксированы)
- **Сданные отчёты** (`status='done'`) — НЕ трогаем; фото и карточки остаются.
- **Уведомлять** затронутых сотрудников — ДА, бот шлёт короткое «задание отменено».
- **После очистки** — сразу пересоздать по расписанию из настроек.
- **Охват** — все АЗС за сегодня.

### Принятые рекомендации (по умолчанию, помечены для ревью)
- **R1 — мягкая отмена, не физическое удаление.** Несданные задания переводятся в новый терминальный статус `cancelled`, а не удаляются из БД. Причина: аудит/история, и старые «временные» слоты (старое время) не должны болтаться как `expired`/открытые в статистике. Из активных списков и KPI `cancelled` скрывается. *(Альтернатива — физический DELETE строк + фото — более разрушительна; не выбрана.)*
- **R2 — пропускать уже сдавших.** При пересоздании АЗС, у которой сегодня есть сданный отчёт (`status='done'`), повторное задание НЕ создаётся (исходим из «один отчёт на АЗС в день»).
- **R3 — предусловие расписания.** Пересоздание берёт времена из настроек (`settings.report.dispatchTimes` / профили). Новое расписание должно быть сохранено в настройках ДО нажатия кнопки (карточка «Расписание рассылки» на дашборде, [reviewer.client.vue:1222](../../../frontend/app/pages/reviewer.client.vue)). В модалке-подтверждении — явная подсказка про это.

## Не-цели (вне объёма)

- Редактор расписания — уже существует (карточка «Расписание рассылки»).
- Ручная рассылка на произвольное время — уже существует («Запросить отчёт вне расписания», [reviewer.client.vue:1314](../../../frontend/app/pages/reviewer.client.vue)).
- Удаление файлов с Диска — папки фото остаются (переиспользуются/безвредны).
- Очистка за прошлые/будущие дни — только «сегодня» (с опциональным явным `date`).

## Терминология (важно: две таблицы)

- **`dispatch_plan`** ([dispatchPlanStore.js](../../../backends/node/api/src/reports/dispatchPlanStore.js)) — ПЛАН: когда разослать. Ключ `plan_date='YYYY-MM-DD'`. Статусы `planned`/`dispatched`/`failed`. Есть `deletePlannedForDate` (удаляет только `planned`).
- **`dispatch_log`** ([dispatchLogStore.js](../../../backends/node/api/src/dispatch/dispatchLogStore.js) запись / [reportsStore.js](../../../backends/node/api/src/reports/reportsStore.js) чтение) — ВЫДАННОЕ ЗАДАНИЕ (строка задачи + карточка CRM `report_item_id`). Ключ `slot_key='YYYY-MM-DD:HHmm'` (или `manual:YYYY-MM-DD:HHmm`). Статусы: `reserved`, `new`, `in_progress`, `done`, `expired`, `failed`. `done` = сдан. Метода удаления нет.

Плановую строку рассылает по слоту по-минутный планировщик ([dispatchScheduler.js](../../../backends/node/api/src/dispatch/dispatchScheduler.js)), а не роут генерации.

## Поведение

### Триггер
Кнопка **«Перевыпустить задания на сегодня»** на дашборде ([reviewer.client.vue](../../../frontend/app/pages/reviewer.client.vue)), рядом с карточками «Расписание рассылки» и «Запросить вне расписания». Видна только админу (право `capabilities.settings`). Открывает модалку-подтверждение с числами (предпросмотр).

### Шаг 0 — Предпросмотр (dry-run)
Перед подтверждением фронт запрашивает предпросмотр: сколько заданий будет снято (`N`), по скольким АЗС (`M`), сколько сданных останется (`K`), сколько АЗС будут пропущены при пересоздании (уже сдали). Модалка: «Снести и пересоздать задания на сегодня по всем АЗС? Будет снято N по M АЗС. Сданные (K) останутся. Пересоздание возьмёт времена из настроек — убедитесь, что новое расписание сохранено».

### Шаг 1 — Очистка (clear)
«Сегодня» = `todayInTz(settings.timezone)` ([reportsRoutes.js:756](../../../backends/node/api/src/reports/reportsRoutes.js)), формат `YYYY-MM-DD`.
- **dispatch_log:** все строки сегодняшнего дня (`slot_key LIKE 'today:%' OR 'manual:today:%'`, исключая `:reminder:`) со статусом ∈ {`reserved`,`new`,`in_progress`,`expired`,`failed`} → перевести в `cancelled`. Строки `done` — пропустить (сданные).
- **dispatch_plan:** `deletePlannedForDate(today)` — снимает ещё-не-разосланные плановые слоты (`planned`). Уже `dispatched` строки остаются как история (их задание снято на стороне dispatch_log).
- **CRM-карточки (best-effort, не блокирует):** для снятых задач карточка отчёта пуста (фото нет). v1: карточки НЕ удаляем (оставляем; они помечены отменёнными на стороне приложения). *Опционально (флаг)* — удалять через новый `deleteCrmItem` под admin-OAuth; делается best-effort и молча пропускается при мёртвом admin-OAuth (BUG-022). Папки Диска не трогаем.

### Шаг 2 — Уведомление
Для каждого затронутого `admin_user_id` (дедуп по пользователю — одно сообщение, даже если снято несколько слотов) — `notificationService.notify({ userId, message, context: req.bitrixContext })` ([notificationService.js:117](../../../backends/node/api/src/notifications/notificationService.js)). Текст: «Задание на фотоотчёт на сегодня отменено — расписание изменилось. Скоро придёт обновлённое задание». Контекст — `req.bitrixContext` (живой OAuth нажавшего админа → бот-канал работает). Сбой отдельного уведомления best-effort, не валит операцию; считаем `notified`/`notifyFailed`.

### Шаг 3 — Пересоздание
`generateDailyPlan({ planDate: today, candidates, settings, planStore: dispatchPlanStore, regenerate: true })` ([dispatchPlanGenerator.js:283](../../../backends/node/api/src/dispatch/dispatchPlanGenerator.js)) — как роут `/plan/generate`. `candidates` = `loadEnabledAzsCandidates(...)` **минус** АЗС, у которых сегодня есть `done` (R2). Новые `planned`-строки подхватит по-минутный планировщик: те, чьё `execute_at` уже прошло — на ближайшем тике, остальные — в своё новое время. Дополнительный мгновенный dispatch не делаем (штатный механизм сам разошлёт).

### Идемпотентность
Повторное нажатие безопасно: снимать уже нечего (всё `cancelled`/`done`), `generateDailyPlan(regenerate)` чистит `planned` и пере-вставляет (upsert `ON CONFLICT DO NOTHING`). Результат тот же.

## API

Новый роут в [reportsRoutes.js](../../../backends/node/api/src/reports/reportsRoutes.js), гард `capabilities.settings` (как настроечные роуты, [server.js:509](../../../backends/node/api/server.js)).

```
POST /api/reports/today/reissue
body: { date?: 'YYYY-MM-DD', dryRun?: boolean }
```
- `dryRun:true` → только посчитать и вернуть числа, ничего не менять.
- Ответ: `{ ok, planDate, affected, azsAffected, submittedKept, cancelled, notified, notifyFailed, regenerated, skippedSubmittedAzs }`.
- Гард: `if (!req.accessContext?.capabilities?.settings) return res.status(403)...`.
- Контексты: `req.bitrixContext` (уведомления, загрузка кандидатов), `getAdminContext()` — только для опц. CRM-удаления.

Фронт ([api.ts](../../../frontend/app/stores/api.ts)): `reissueTodayTasks({ date?, dryRun? })`.

## Обработка ошибок и устойчивость

- Порядок: посчитать → cancel (БД, транзакция/один UPDATE) → notify (best-effort, по одному) → regenerate (БД). При обрыве на notify/regenerate — задания уже сняты, сотрудник не «висит» на старом; повторный клик до-пересоздаст.
- CRM-удаление и каждое уведомление — best-effort; ошибки собираются в счётчики, не валят 500.
- Таймзона — строго `settings.timezone` (как генератор плана), иначе «сегодня» разъедется.
- Гонка с планировщиком: между cancel и regenerate тик может разослать только что вставленные `planned`-строки — это и нужно. Дублей нет за счёт `UNIQUE(slot_key, azs_id)` и upsert.
- Зависимость: бот-доставка уведомлений живёт под OAuth нажавшего админа (живой) — не упирается в BUG-022; при `BITRIX_BOT_MODE≠bot` уйдёт `im.notify` (без кнопок) — приемлемо для текста-отмены.

## Доступ

Только админ — `capabilities.settings`. Совпадает с правом менять расписание (логично: кто меняет расписание — тот и перевыпускает).

## Тестирование

- `cancelNotSubmittedForDate`: трогает только сегодняшние не-`done`; `done` остаются; статус → `cancelled`; reminder-слоты не задеты.
- dry-run: ничего не меняет, числа верны.
- Уведомления: один месседж на пользователя; best-effort при сбое.
- Регенерация: создаёт по `dispatchTimes`; АЗС со сданным сегодня — пропущены (R2).
- Идемпотентность повторного клика.
- Таймзона (граница суток).
- Гард: проверяющий/AZS-админ → 403; админ → 200.
- `getSummary`: `cancelled` не в `open`/`overdue`.
