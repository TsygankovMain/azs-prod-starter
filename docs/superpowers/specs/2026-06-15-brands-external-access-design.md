# Спека: «Бренды» — внешний доступ к фото по бренду (штатный шэр Bitrix Диска)

**Версия:** 1.0  
**Дата:** 2026-06-15  
**Статус:** Согласован дизайн; реализация — следующий спринт  
**Авторы:** Продукт — АЗС, техническое оформление  

---

## 1. Проблема и цель

### Что болит

Клиент ГПН (регион Москва) хочет просматривать фото **своих 11 АЗС**, не имея доступа к внутреннему порталу Bitrix24. Раньше фото хранились на Google Drive и были напрямую доступны клиенту. После миграции операторы грузят фото в Bitrix Диск, но механизма дать ГПН доступ только к «своим» АЗС нет — нужно либо давать доступ ко всему хранилищу (небезопасно), либо вручную копировать (трудоёмко).

### Цель

Дать администратору возможность объединить набор АЗС в «бренд», создать для него отдельную папку на Bitrix Диске и получить публичную внешнюю ссылку на эту папку. Новые фото АЗС бренда автоматически попадают в папку бренда. Ссылка передаётся ГПН — они открывают в браузере и видят только фото своих АЗС.

### Временной контекст

> ⚠️ **Дедлайн «завтра» (2026-06-16) — СТОПГАП:** полноценная фича (бренды в БД + роутинг + UI) займёт ~4–5 дней разработки. Для «завтра» возможен **ручной стопгап**: администратор вручную создаёт папку в Bitrix Диске, вручную копирует туда нужные фото из папок 11 АЗС и вручную получает публичную ссылку через нативный UI Диска. Это не требует изменений в коде и закрывает срочную потребность. Полноценная автоматизация — в отдельном спринте (см. раздел 11).

## 2. Как работает сейчас

### 2.1 Загрузка фото (upload-путь)

**Файл:** `backends/node/api/src/reports/reportsRoutes.js`

Маршрут `POST /:id/photo` (обработчик ~строка 1440):

1. Оператор загружает файл через мобильное приложение.
2. Вызывается `uploadPhoto(bitrixClient.diskApi, { rootFolderId, azsId, azsName, slotDate, slotHHmm, ..., folderNameTemplate })` — строка **1477–1490**.
3. Функция `uploadPhoto` из `backends/node/api/src/disk/diskService.js`:
   - строка **312**: `buildFolderPath({ capturedAt: folderDate, azsId, azsName, folderNameTemplate })` — строит путь по шаблону.
   - строка **313**: `ensureFolderPath(diskApi, { rootFolderId, path: folderPath })` — создаёт вложенные папки через `disk.folder.addsubfolder` (строка **675** в `bitrixRestClient.js`).
   - строка **315**: `uploadFileReplacingDuplicate(diskApi, { folderId: targetFolderId, ... })` — загружает файл через `disk.folder.uploadfile` (строка **693** в `bitrixRestClient.js`).
4. Возвращает `{ folderId: targetFolderId, fileId, diskObjectId, ... }`.
5. Строка **1492–1501**: `upsertPhoto({ reportId, photoCode, fileId, diskFolderId: uploaded.folderId, diskObjectId, ... })` — `diskFolderId` сохраняется в таблицу `report_photo.disk_folder_id`.
6. Строка **1514–1523**: ставится задание в очередь `crmSyncJobStore` с `diskFolderId: uploaded.folderId`.

## 3. Модель бренда в БД

### 3.1 Таблицы

#### Таблица `brand`

| Колонка | Тип | Описание |
|---------|-----|---------|
| `id` | `BIGSERIAL` (PG) / `BIGINT AUTO_INCREMENT` (MySQL) | Первичный ключ |
| `name` | `TEXT NOT NULL` | Название бренда (например, «ГПН Москва») |
| `disk_folder_id` | `BIGINT NULL` | ID папки бренда на Bitrix Диске |
| `disk_folder_path` | `TEXT NULL` | Путь папки (для справки/отладки) |
| `external_link` | `TEXT NULL` | Последняя сохранённая внешняя ссылка |
| `external_link_updated_at` | `TIMESTAMPTZ NULL` | Когда ссылка была получена/обновлена |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | |
| `updated_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | |

#### Таблица `brand_azs`

| Колонка | Тип | Описание |
|---------|-----|---------|
| `id` | `BIGSERIAL` / `BIGINT AUTO_INCREMENT` | Первичный ключ |
| `brand_id` | `BIGINT NOT NULL` | FK → `brand.id` |
| `azs_id` | `TEXT NOT NULL` | ID АЗС из CRM (строка, как везде в системе) |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | |

Ограничение: `UNIQUE(azs_id)` — одна АЗС принадлежит максимум одному бренду.

### 3.2 Инвариант «одна АЗС = один бренд»

- **БД**: `UNIQUE(azs_id)` на таблице `brand_azs` обеспечивает инвариант на уровне базы.
- **UI**: при добавлении АЗС в бренд — мультиселект показывает уже занятые АЗС серыми/disabled (аналогично профилям рассылки, `dispatch-profiles-spec.md` раздел 3.3).
- **API**: при попытке добавить уже занятую АЗС — бэкенд возвращает 409 с указанием, в каком бренде она уже числится.

### 3.3 Резолвинг azsId → бренд

При загрузке фото (upload-путь, раздел 2.1) нужно найти бренд для данной АЗС:

```
function resolveBrandForAzs(azsId):
  SELECT brand_id FROM brand_azs WHERE azs_id = azsId LIMIT 1
  → если есть: вернуть brand (JOIN brand)
  → если нет: вернуть null (АЗС не в бренде, роутинг без изменений)
```

Запрос простой по индексу — производительность не является проблемой даже при высокой частоте загрузок.

## 4. Папка бренда и роутинг новых фото

### 4.1 Папка бренда на Диске

При создании бренда администратор выбирает один из двух вариантов:

**Вариант 4.1A (создать новую):** приложение вызывает `disk.folder.addsubfolder` (`bitrixRestClient.js` строка **675**) внутри корневой папки приложения `AZS-Photo-Reports/`. Имя папки = название бренда (санитизированное). ID возвращается и сохраняется в `brand.disk_folder_id`.

**Вариант 4.1B (указать существующую):** администратор вводит ID существующей папки Диска вручную. Проверяется доступность папки вызовом `disk.folder.getchildren` (строка **649**). ID сохраняется в `brand.disk_folder_id`.

> ⚠️ ОТКРЫТЫЙ ВОПРОС: Нужен ли UI для ввода существующего ID папки в MVP? Или достаточно только «Создать новую»? Рекомендуется оставить оба варианта, чтобы не блокировать ручной стопгап.

### 4.2 Структура папок с брендом

**Текущая структура:**

```
AZS-Photo-Reports/
  2026-06/
    15/
      42_Лукойл-Тверская/
        АЗС-Тверская_2026-06-15_0900_фасад.jpg
```

**Новая структура (папка бренда как контейнер):**

```
AZS-Photo-Reports/
  ГПН-Москва/                           ← папка бренда (brand.disk_folder_id)
    42_Лукойл-Тверская/                 ← папка АЗС
      2026-06/                          ← месяц
        15/                             ← день
          АЗС-Тверская_2026-06-15_...jpg
  Другой-Бренд/
    ...
  2026-06/                              ← АЗС без бренда — по старому шаблону
    15/
      ...
```

**Новый `folderNameTemplate` для АЗС в бренде:**

```
{azs}_{azs_name}/{yyyy-mm}/{dd}
```

При upload-е, если АЗС принадлежит бренду:
- `rootFolderId` подменяется на `brand.disk_folder_id` (вместо глобального `settings.disk.rootFolderId`)
- `folderNameTemplate` подменяется на `{azs}_{azs_name}/{yyyy-mm}/{dd}`

Функция `uploadPhoto()` (`diskService.js` строки **287–329**) уже принимает `rootFolderId` и `folderNameTemplate` параметрами — подмена происходит в слое маршрутов (`reportsRoutes.js` строка **1477–1490**) до вызова `uploadPhoto`.

### 4.3 Влияние на поле «Папка Диска» в CRM и /resync

После внедрения брендов `uploaded.folderId` (строка **1497**) будет указывать на папку внутри бренда (`ГПН-Москва/42_Лукойл-Тверская/2026-06/15/`), а не на старую структуру. Это значение записывается в `report_photo.disk_folder_id` и далее в CRM-поле через `reportCrmSync.js` строка **27–28**.

**Поведение /resync:** при ресинке (`reportsRoutes.js` строка **1697**) берётся `diskFolderId` из фотографий — это будет папка внутри бренда. Это корректно: CRM-карточка будет показывать ссылку на конкретную дневную папку АЗС внутри бренда, что правильно для операционной работы. Ссылка на всю папку бренда (для передачи ГПН) хранится отдельно в `brand.external_link`.

## 5. Внешняя ссылка

### 5.1 Получение ссылки

Кнопка «Получить/обновить ссылку» в UI бренда → бэкенд вызывает REST-метод Bitrix:

```
disk.folder.getexternallink({ id: brand.disk_folder_id })
```

> **Примечание:** метод `disk.folder.getexternallink` **генерирует** публичную ссылку. Если ссылка уже существовала — возвращает её же. Метод не управляет паролем или сроком действия.

Ссылка сохраняется в `brand.external_link` + `brand.external_link_updated_at`.

### 5.2 Отображение ссылки

В карточке бренда показывается поле «Внешняя ссылка» с кнопкой «Скопировать». Рядом — кнопка «Обновить ссылку» (вызывает `disk.folder.getexternallink` повторно, сохраняет актуальный URL).

### 5.3 Ограничение: отзыв ссылки

> **ОГРАНИЧЕНИЕ (зафиксировано):** REST API Bitrix24 не предоставляет метода для **отзыва** публичной ссылки на папку. Отзыв (деактивация) доступен только через нативный UI Bitrix Диска вручную. Приложение показывает об этом предупреждение в UI бренда: «Для деактивации ссылки — откройте папку в Bitrix Диске и отключите публичный доступ вручную».

## 6. Пароль — Вариант A (штатный, вручную в Диске)

**ПРИНЯТОЕ РЕШЕНИЕ — ВАРИАНТ A:**

REST API Bitrix24 (`disk.folder.getexternallink`) возвращает только URL публичной ссылки. Метод **не принимает** параметры пароля или срока действия — они отсутствуют в API.

Если тариф Bitrix24 портала поддерживает защиту публичных ссылок паролем или ограничение по сроку — администратор настраивает это **вручную** в нативном интерфейсе Bitrix Диска:
1. Открыть папку бренда в Диске.
2. Нажать «Поделиться» → «Публичная ссылка» → «Настройки».
3. Установить пароль и/или срок действия.

Приложение не управляет паролем, не хранит его и не показывает. В UI бренда — информационный блок: «Пароль и срок действия — настраиваются вручную в Bitrix Диске».

> ⚠️ **ОГРАНИЧЕНИЕ:** Если тариф Bitrix24 не поддерживает пароль на ссылку — пароля нет. Это ограничение платформы, а не приложения.

## 9. Риски и граничные случаи

### 9.1 Смена бренда у АЗС

Если АЗС переводится из одного бренда в другой (или выходит из бренда), **исторические фото** (до смены) остаются в старой папке бренда. Новые фото (после смены) идут в новую папку. Это нормальное поведение, но:

> ⚠️ Внешняя ссылка на старую папку бренда продолжает работать и показывает фото до смены. Это может быть нежелательно. Решение: документировать поведение в UI («При смене бренда исторические фото не перемещаются»). Автомиграция — вне scope.

### 9.2 Публичная ссылка без пароля

Если администратор не установил пароль вручную в Диске — ссылка открыта без аутентификации. Любой, у кого есть URL, видит фото. Это может быть неприемлемо для некоторых клиентов. Документировать в UI.

### 9.3 `/resync` при фото в разных папках

Если у отчёта фото из разных папок (теоретически возможно при ошибках или миграции), `/resync` берёт первый ненулевой `diskFolderId` (`reportsRoutes.js` строка **1697**). После внедрения брендов — все фото одного отчёта гарантированно в одной папке (один отчёт = одна АЗС = один бренд). Риск остаётся только для переходного периода и исторических данных.

### 9.4 Согласованность папок (папка не удалена из Диска)

`brand.disk_folder_id` хранит Bitrix ID папки. Если папка удалена в Диске, загрузки в неё будут завершаться ошибкой. Требуется обработка ошибки `disk.folder.uploadfile` → 404/error — с понятным сообщением администратору.

### 9.5 «Ровно эти АЗС» для ГПН

Публичная ссылка даёт доступ к **папке бренда целиком**. Если в папке бренда есть АЗС, которые не должны быть видны конкретному получателю ссылки — это необходимо решать другим способом (создать отдельный бренд, создать отдельную ссылку). Это ограничение модели: одна ссылка = один бренд = набор АЗС.

| BL-7 | Несколько ссылок / разные права для разных получателей | Бэклог (требует отдельных папок под каждого получателя) |
