- Дата: 2026-05-31
- Ветка реализации: `feature/v2.0` (от `master` @ `2e60861`)

## 1. Цель и контекст

Приложение работает в проде. Накопились замечания клиента из чата и один P0-риск из ревью (потеря CRM-синка при рестарте). v2.0 = облагораживание всех экранов на B24 UI Kit + мобильная адаптация экрана администратора + точечные фичи из переписки + закрытие P0-риска. Стиль и архитектурные решения подтверждены через визуальные мокапы.

## 2. Объём (что входит)

Сгруппировано по спринтам. Приоритеты: **P0** (риск данных) → **P1** (UX, мешает работе) → **P2** (фичи-улучшения) → **P3** (за флагом).

### Спринт 1 (P0) — Надёжность данных
1. **Durable CRM-sync очередь.** Сейчас `reportCrmSyncQueue` живёт в памяти (`reportsRoutes.js:613`); рестарт/деплой между загрузкой фото (ответ 200 уже отдан) и синком в CRM теряет синк → осиротевший элемент смарт-процесса, `synced=false` невидим, без ретрая (`reportsRoutes.js:1025-1033`). Делаем:
   - персист очереди синков в БД (таблица `crm_sync_jobs`: reportId, payload, attempts, status, lastError, nextAttemptAt);
   - воркер забирает из БД, ретраит с backoff `[800,1600,3200]`+jitter (поведение уже есть — переносим на durable-store);
   - после рестарта pending-джобы возобновляются;
   - в карточке отчёта хранится `synced: boolean` + `lastSyncError`;
   - на экране проверяющего бейдж `не синхронизировано` + ручная кнопка «Пересинхронизировать».
2. **Выравнивание folder-template defaults.** Код-дефолты в `defaultSettings.js:43` и `diskService.js:1` всё ещё старые (`{yyyy-mm}/{dd}/{azs}_{azs_name}`). В проде переопределено через настройки, но reset/reinstall вернёт регрессию (#4). Привести оба дефолта к согласованному формату (тот же, что сейчас в проде).

### Спринт 2 (P1) — Экран администратора, мобильный фокус-режим (A1)
Рефакторинг **на месте** `frontend/app/pages/admin/[reportId].client.vue` (не переписывать с нуля — сохранить рабочую логику очереди x2/x1, камеры, retry):
- **Фокус-режим:** один слот = один экран. Камера на весь кадр, крупная кнопка «Сделать фото» (тач ≥44px).

- Тяжёлые блоки вынести в под-компоненты (`CameraSlot`, `SlotProgress`, `SlotsQueue`) для читаемости.

## 4. Архитектура изменений (по слоям)

### Frontend
- Все экраны → B24 UI Kit, единые статус-бейджи, отступы/радиусы (air), empty states.
- `admin/[reportId].client.vue` → рефактор на месте + под-компоненты камеры/слотов.
- `reviewer.client.vue` → миграция на B24, мультивыбор, фильтр+KPI по АЗС, хинт расписания, бейдж/кнопка ресинка.
- `settings.client.vue` → адаптивная навигация (desktop side-nav / mobile accordion), подсказки, статусы.
- `index.client.vue`, `install.client.vue` → статус-шаги.

### Backend
- Новая таблица/стор `crm_sync_jobs` + воркер durable-очереди (переиспользует существующий backoff/jitter из `reportCrmSync.js`).
- `reportsRoutes.js`: запись `synced`/`lastSyncError` в отчёт; эндпоинт ручного ресинка `POST /api/reports/:id/resync`; ответ `/photo` без изменения контракта (`syncQueued:true` остаётся).
- `defaultSettings.js` + `diskService.js`: выравнивание folder-template default.
- `dispatchService.js`: опциональный `keyboard` для deep-link за `ENABLE_REPORT_DEEP_LINK`.
- Опционально: guard-лог адресации в `dispatchScheduler`/`notificationService`.

### Data flow (без изменения внешних контрактов)
- Загрузка фото: `POST /photo` → Disk + DB (синхронно, ответ 200 + `syncQueued:true`) → **durable** crm_sync_job (вместо in-memory) → воркер → CRM. При краше джоб переживает рестарт.
- Отчёт по АЗС: фронт прокидывает `azsIds` в существующие `/api/reports` и `/api/reports/summary`.

## 5. Обработка ошибок
- CRM-sync: ретраи с backoff+jitter, после N попыток статус `failed` + `lastSyncError`, видно проверяющему + кнопка ручного ресинка. Никаких «тихих» потерь.
- Камера/загрузка фото: существующая логика x2→x1 и retry сохраняется; ошибочные слоты подсвечиваются, можно переснять/повторить.
- Служебные экраны: B24Alert с понятным текстом при сбое install/route.

## 6. Тестирование
- Backend unit: durable очередь (enqueue/persist/resume-after-restart/backoff/fail), folder-template default, deep-link keyboard за флагом. Запуск `cd backends/node/api && pnpm test` — зелёным, плюс существующие.
- Smoke: загрузка фото → kill процесса до синка → рестарт → джоб довершился; `synced=false` виден и ручной ресинк работает.
- Frontend: ручная проверка в Bitrix24 iframe — мобильный фокус-режим админа, мультивыбор+фильтр АЗС у проверяющего, навигация настроек (desktop/mobile).
