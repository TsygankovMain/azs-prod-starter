# Спека: Фикс-спринт 5 «Канал бота + добивание смоука»

**Статус:** на утверждении продакта · **Дата:** 2026-06-11 · **Источник:** bug-backlog (BUG-009…018, диагнозы подтверждены логами/кодом в сессии смоука 2026-06-11) · **Приоритет продакта:** блокирующие → канал бота (018/009) → HIGH (013, 014) → MEDIUM (011, 016, 017) → LOW/PERF (012, 015, 010).

## 1. Цель

Вернуть штатный канал уведомлений (сообщения бота с кнопками вместо тихого пуша), добить найденные смоуком дефекты фотоленты и отчётов, закрыть гигиену (ошибки, логи, навигация) и ускорить выборки из Битрикс24. Один спринт, релизы — по волнам (см. §5).

## 2. Не-цели (вне скоупа)

- Новые фичи (V2 «Принято в работу», серверные thumbnails) — отложенное по желанию.
- Полная ре-архитектура crmSync — BUG-010 ограничен пагинацией клиента (см. W4-1).
- Проектный lint-долг вне зон спринта (отдельная зачистка, кандидат хвоста спринта при свободном агенте).

## 3. Волна 1 — канал бота (BUG-018 + BUG-009, HIGH, первый релиз)

### W1-1 · Формат клавиатуры (корень PARAM_KEYBOARD_ERROR)
**Файлы:** `src/dispatch/dispatchService.js`, `src/dispatch/timeoutWatcher.js`, тесты dispatchDeepLink/timeoutWatcher/notificationService.
**Фикс:** структура keyboard по официальной доке (apidocs → chat-bots-v2 → message-keyboards): ПЛОСКИЙ массив кнопок, перенос строки — элемент `{TYPE:'NEWLINE'}`; рекомендованная обёртка `{BOT_ID, BUTTONS:[…]}` (BOT_ID уже известен notificationService). Никаких вложенных массивов-строк.
**Приёмка:** юнит-тест структуры (плоский массив/обёртка, NEWLINE между кнопками задания); на смоуке задание приходит СООБЩЕНИЕМ бота с кнопками «Открыть приложение» и «Не успеваю — указать причину», просрочка — с «Указать причину».

### W1-2 · Видимый fallback вместо тихой деградации
**Файлы:** `src/notifications/notificationService.js`, `src/dispatch/dispatchService.js`/`timeoutWatcher.js` (точки вызова), `dispatchLogStore` (только запись error_text), тесты.
**Фикс:** при падении бот-отправки и уходе в `im.notify.personal.add`: (а) `logger.warn('bot_channel_degraded', {reason})` с полным текстом ошибки Bitrix; (б) в `dispatch_log.error_text` дописывать `delivered via notify fallback: <reason>` при УСПЕШНОМ нотифай-фоллбеке (статус остаётся успешным — доставка была, но деградированная; отображение см. W3-1).
**Приёмка:** тест «бот упал → notify ушёл → error_text содержит пометку, warn залогирован»; канал бота больше не умирает незаметно.

### W1-3 · Добивание зависших слотов + честная ручная рассылка
**Файлы:** `src/dispatch/dispatchScheduler.js`, роут ручной рассылки (reportsRoutes/dispatchRoutes — по факту), тесты.
**Фикс:** (а) слоты «Запланирован», чей scheduled_at просрочен более чем на N минут (константа, дефолт 30), при очередном тике либо исполняются (если контекст жив), либо переводятся в `failed` с error_text `skipped: no auth context at send time` — вечных «Запланирован» не остаётся; (б) ручная рассылка при недоступном auth-контексте/боте отвечает 503 с errorCode `BOT_UNAVAILABLE` (+ словарь useErrorText), а не молча создаёт зависающий слот.
**Приёмка:** тесты обоих путей; в тех-таблице не может существовать «Запланирован» старше N минут после срока.

### W1-4 · Self-heal бота (страховка на будущее)
**Файлы:** `server.js` (ручка по образцу refresh-avatar), `src/notifications/botRegistryService.js` (переиспользовать ensureBot), `settings.client.vue` (кнопка в настройках, секция бота), тесты.
**Фикс:** `POST /api/admin/bot/reregister` (роль settings) → ensureBot с принудительной перерегистрацией; кнопка «Перерегистрировать бота» в настройках с подтверждением (useConfirm) и тостом результата. Авто-self-heal в отправке: при ошибке вида BOT_ID/NOT_FOUND — один ensureBot + один повтор (не для PARAM_*-ошибок).
**Приёмка:** тесты ручки (роль, успех, ошибка Bitrix); кнопка работает на смоуке.

### W1-Env (продакт, параллельно, без кода)
Перевбить `CLIENT_ID`/`CLIENT_SECRET` в панели Timeweb со страницы приложения, перезапуск; критерий — в логах исчезает `wrong_client` (контрольный экспорт логов через час простоя). Двухфакторная проверка волны: после релиза W1 и env-фикса — рассылка работает после >1 часа простоя без входа админа.

## 4. Волна 2 — HIGH-дефекты фотоленты и отчётов (сначала диагностика, потом фикс)

### W2-0 · Диагностика BUG-013 (превью) и BUG-014 (карточка) — до фиксов, строго
**BUG-013:** инструментованный воспроизводящий запрос: статус/тело `GET /api/reports/photos/:r/:c/preview` на проде (просим у продакта Network-факт; параллельно — чтение пути в коде с фейками). Развилки: (а) 502 preview_failed → чинить серверный путь скачивания (downloadFileContent/disk_object_id/контекст — по фактической ошибке); (б) 404/маршрут → монтаж/порядок роутеров; (в) 200, но blob не рисуется → фронт (создание objectURL/CSP в iframe).
**BUG-014:** Network-факт запроса карточки (`reports?azsId=…` — какой azsId ушёл, что вернулось) ИЛИ воспроизведение на dev-портале. Развилки: (а) id-пространства разошлись (выпадашка vs dispatch_log.azs_id) → нормализовать источник опций карточки на тот же id, что пишет рассылка (+тест-инвариант «карточка видит то же, что витрина»); (б) фильтр дат/выборка → фикс стора + тест.
**Правило:** фикс-задача стартует только с подтверждённым корнем (Iron Law systematic-debugging). Если прод-факты не получены к старту волны — диагностический агент воспроизводит на dev-портале самостоятельно.

### W2-1 · BUG-013 фикс по результату + UX ошибок превью
**Файлы:** по развилке + `PhotoFeedGrid.vue`/`PhotoLightbox.vue`/`RemarkJournal.vue` (общий паттерн тайла).
**Фикс UX (безусловно):** previewErrors → видимое состояние тайла («не удалось», иконка) + повторная попытка по тапу; revoke-логика не трогается.
**Приёмка:** превью видны на проде (витрина и лента); сломанное превью показывает состояние и ретраится.

### W2-2 · BUG-014 фикс по результату
**Приёмка:** карточка Тестовой показывает сегодняшнюю сдачу и вчерашний провал; регресс-тест инварианта с витриной.

## 5. Волна 3 — MEDIUM (гигиена доверия)
