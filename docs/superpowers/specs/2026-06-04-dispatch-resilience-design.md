# Дизайн: переживаемость редеплоя для авто-генерации плана рассылки

- Дата: 2026-06-04
- Ветка реализации: `feature/dispatch-resilience` (от `master`)
- Критичность: ВЫСШАЯ — основной прод-путь. Внедрять аккуратно, тесты + ревью.
- Контекст: рандомизированный план рассылки (`dispatch_plan`) уже в проде (ON по умолчанию). Проблема: после ночного редеплоя план не сгенерировался — авто-генерация и поминутный исполнитель завязаны на admin OAuth-контекст из файла `./data/auth-context.json`, который (а) теряется при редеплое и (б) наполняется только когда админ откроет приложение. БД на проде в одном контейнере — тоже очищается при редеплое.
- Источники: разбор `dispatchScheduler.js`, `bitrixRestClient.js`, `bitrixAppSettingsStore.js`, `authContextStore.js`, `tokenRefreshScheduler.js` + прод-факт (psql: план на сегодня есть после ручной кнопки, но авто-путь молчал).

## Цель

Чтобы авто-генерация И рассылка работали после редеплоя БЕЗ участия человека и БЕЗ потери данных. Ручная кнопка «Сформировать график» остаётся как fallback.

## Корень (два разрыва)

1. **Токен.** `getRuntimeContext()` (и в генерации, и в исполнителе `executeDuePlans`, и в чтении настроек) берёт токен из `authContextStore.getLastAdminContext()` → файл, теряется/пустой после редеплоя. Гард `if (!context.authId) return` → молчаливый скип.
2. **Хранение плана.** `dispatch_plan` только в БД → очищается при редеплое.

## Решение — 3 связанные секции

### Секция 1 — Вебхук как серверный токен для фоновых задач ( APPROVED)
- В портале Bitrix создаётся ВХОДЯЩИЙ вебхук (scope: crm, im, imbot, disk, user). Даёт URL `https://<portal>/rest/<user_id>/<code>/` с НЕистекающим токеном.
- В `.env`: `BITRIX_WEBHOOK_URL=...` (один секрет).
- `bitrixRestClient`: новый режим «webhook» для фоновых вызовов. Контекст-резолюция для фона: если задан webhook → строить runtime из него (endpoint = webhook base, без `auth` параметра — вебхук авторизует сам по URL). Пользовательские запросы (экраны, кнопки) продолжают идти через per-request OAuth `req.bitrixContext` как сейчас.
- Новый модуль `src/auth/webhookContext.js`: `buildWebhookRuntime(webhookUrl)` → `{ endpoint, isWebhook:true }`. Чистый, тестируемый.
- Фоновые потребители (генерация плана, исполнитель, чтение/запись app.option в фоне) получают контекст через новый `getBackgroundContext()`: **webhook, если задан; иначе фоллбэк на getLastAdminContext (как сейчас)**. Обратная совместимость: без webhook — поведение прежнее.
- **Почему закрывает корень:** вебхук не истекает, не зависит от сессий/редеплоя. После редеплоя сервер сразу работает с Bitrix.
- **Trade-off:** вебхук под одним пользователем-владельцем (стабильные права — как раньше strict-admin). Допустимо для серверной автоматики.
- **Замечание REST-формата:** при вебхуке URL уже содержит user_id+code; метод вызывается как `<webhookUrl><method>.json` БЕЗ `auth` в теле. bitrixRestClient.callInternalOnce должен НЕ добавлять `auth`, когда контекст webhook. Проверить и в callWithTransientRetry/refresh — для webhook рефреш OAuth НЕ применяется (нет refresh_token; ошибки вебхука не refreshable).

### Секция 2 — План в Bitrix24 app.option (durable-зеркало) (APPROVED)
- Механизм как у настроек: `app.option.set`/`get` (см. `bitrixAppSettingsStore.js`). Ключ `azs_dispatch_plan_v1`.
- Модель «БД = рабочая копия, Bitrix = сейф»:
  - При генерации: план пишется в БД (`dispatchPlanStore`) И в app.option (зеркало JSON `{ planDate, rows:[{azsId, adminUserId, baseTime, executeAt(ISO), jitterMinutes, status}] }`).
  - При старте/чтении плана: если в БД нет плана на сегодня, но в app.option есть → **rehydrate** (восстановить строки в БД, идемпотентно через upsertPlanned). Лог `dispatch_plan.rehydrated_from_bitrix`.
  - Статусы (dispatched/failed) пишутся в БД; зеркало в app.option обновляется периодически/при изменениях (не критично к секунде — второй рубеж от дублей это `UNIQUE(slot_key,azs_id)` в dispatch_log).
- Новый модуль `src/reports/dispatchPlanMirror.js`: `writeMirror({context, planDate, rows})`, `readMirror({context})`, `rehydrateIfEmpty({context, planDate, planStore})`. Зависит от bitrixClient + dispatchPlanStore. Изолирован, тестируется с фейками.
- Объём: ~71 строка × ~6 полей ≈ 10–15 КБ JSON — app.option держит.
- **Зависит от секции 1:** чтение/запись app.option в фоне требует токен → webhook.

### Секция 3 — Ретрай генерации + алерт вместо буфера «завтра» (APPROVED)
- НЕ делаем двухдневную модель (риск рассинхрона «показываем сегодня / под капотом завтра» + регенерация при смене настроек). Вместо неё:
- **Ретрай генерации:** планировщик при тике проверяет — есть ли план на сегодня (БД или зеркало). Если нет И сейчас в пределах рабочего окна (или до него) → пытается сгенерировать. Повторяется каждый тик, пока план не построится (идемпотентный upsert не задвоит). Первый доступный момент (webhook есть всегда → практически сразу на старте) строит план.
- **Алерт:** если плана на сегодня нет к моменту `workWindow.start` (или к первому base-времени) → уведомление в бот проверяющему(-им): «План на сегодня не сформирован, откройте приложение и нажмите "Сформировать график"». Через существующий `notificationService`. Один раз в день (флаг «alert_sent» в зеркале/БД).
- **Почему лучше буфера:** проще, нет дублирования дат, нет риска рассинхрона; ретрай + webhook практически гарантируют генерацию; алерт убирает «молчаливый» провал.

## Архитектура изменений (по слоям)

### Backend
- `src/auth/webhookContext.js` (новый) — построение webhook-runtime.
- `bitrixRestClient.js` — поддержка webhook-контекста (без `auth` в теле, без OAuth-refresh для webhook).
- `server.js` — собрать `getBackgroundContext()` (webhook→fallback admin), прокинуть в scheduler (генерация/исполнитель) и в mirror.
- `dispatchScheduler.js` — генерация/исполнитель используют `getBackgroundContext` вместо строгого admin-only; ретрай генерации в тике; алерт при отсутствии плана.
- `src/reports/dispatchPlanMirror.js` (новый) — зеркало плана в app.option + rehydrate.
- Интеграция rehydrate: на старте плана / в GET /plan, если БД пуста.

### Конфиг
- `.env.example`: `BITRIX_WEBHOOK_URL=` (пусто по умолчанию; без него — старое поведение OAuth-fallback). Документировать создание вебхука.

### Тесты
- `webhookContext`: построение runtime, без auth-параметра.
- `bitrixRestClient`: webhook-режим — вызов идёт на webhook URL без `auth`; OAuth-refresh НЕ триггерится для webhook-ошибок.
- `dispatchPlanMirror`: write/read/rehydrate (фейк bitrixClient + фейк planStore) — rehydrate наполняет пустую БД из зеркала идемпотентно.
- scheduler: генерация под webhook-контекстом (без admin файла); ретрай — второй тик достраивает план, если первый не смог; алерт шлётся один раз при отсутствии плана.
- Регресс: без webhook (BITRIX_WEBHOOK_URL пуст) — поведение прежнее, существующие тесты зелёные.

## Обработка ошибок
- Нет webhook И нет admin-контекста → как сейчас (скип + алерт по секции 3). Не хуже текущего.
- app.option недоступен при записи зеркала → лог, не валит генерацию (БД — основная). При чтении недоступен → работаем по БД.
- Вебхук невалиден/отозван → ошибки не refreshable; лог `webhook_invalid`; алерт. Чинится заменой `BITRIX_WEBHOOK_URL`.

## Безопасность внедрения
- Всё аддитивно и за наличием `BITRIX_WEBHOOK_URL`. Пустой webhook → старое поведение (нулевой риск регресса при деплое без вебхука).
- Ручная кнопка «Сформировать график» остаётся fallback на весь период.
- Пуш в git — ТОЛЬКО по команде пользователя.

## Подтверждённые решения
- Токен: входящий вебхук (секция 1).
- Хранение плана: Bitrix app.option, БД = рабочая копия (секция 2).
- Буфер «завтра»: НЕ делаем; вместо — ретрай генерации + алерт (секция 3).
- Приоритет: переживаемость редеплоя.
