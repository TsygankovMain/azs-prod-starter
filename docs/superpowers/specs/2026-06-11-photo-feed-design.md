# Спека: Фотолента с замечаниями («рабочий стол проверяющего»)

**Статус:** на утверждении продакта · **Дата:** 2026-06-11 · **Источник требований:** ТЗ заказчика п.4 «Фильтрация и поиск для последующего анализа» + продуктовая проработка с продактом (мокап утверждён в сессии 2026-06-11).

---

## 1. Цель

Офис (проверяющий, админ) быстро находит нужные фото по фильтрам, просматривает их лентой со свайпами и в момент обнаружения проблемы отправляет замечание ответственному за АЗС — с текстом и самими фото. Вся история замечаний централизована и видна всем проверяющим и админам.

## 2. Персоны и доступ

| Роль | Доступ |
|---|---|
| Проверяющий | Полный: лента, отметки, отправка, журнал |
| Админ | Полный (тот же рабочий стол) |
| Оператор/админ АЗС | НЕ видит ленту и журнал; получает замечания личным сообщением от бота |

## 3. Ключевые сценарии (утверждены на мокапе)

**С1 — «Нашёл — отправил» (основной).** Проверяющий открывает ленту (дефолт: «Сегодня · все АЗС · все категории»), листает фото в полноэкране свайпами, на проблемном жмёт «Отметить». Снизу выезжает панель-черновик: получатель «Управляющему» (по умолчанию), кнопки-шаблоны («Переделайте выкладку промо-товара», «Перегрузите правый монитор — старая реклама»), поле текста, «Отправить (1)». Отправил — тост успеха, бейдж «✓ замечание» на фото, запись в журнале. Листает дальше.

**С2 — Несколько проблем на одной АЗС.** Не отправляя панель, отмечает ещё фото ТОЙ ЖЕ АЗС — они копятся в панели («Отправить (3)»). Управляющий получает ОДНО сообщение: текст + 3 фото.

**С3 — Граница АЗС.** Замечание строго в рамках одной АЗС. Тап «Отметить» на фото другой АЗС при непустой панели → мягкая блокировка: «В замечании уже 2 фото АЗС №3» с кнопками «Отправить текущее» и «Очистить и начать с этого фото». Потери выделения молча не бывает.

**С4 — Разбор по журналу.** Админ открывает вкладку «Журнал», переключает вид «По АЗС» → «АЗС №7 — 3 замечания за период», раскрывает, тапает миниатюру → открывается то самое фото в просмотре.

**С5 — Управляющий не заполнен.** В карточке АЗС нет управляющего → панель сразу сообщает «У АЗС №3 не указан управляющий» и предлагает выбрать «Администратору АЗС». Падений при отправке из-за этого не бывает.

## 4. Поведение экранов

### 4.1 Лента (вкладка «Фотолента»)
- Фильтры: период (Сегодня/Вчера/Неделя/Диапазон), мультиселект АЗС, чипы категорий, чип «Замечания: все / с замечаниями / без» (циклический), тумблер «По АЗС» (группировка секциями).
- Дефолт: Сегодня · все АЗС · все категории · поток по времени (новые сверху). Последние выбранные фильтры и тумблер запоминаются между заходами; «Сбросить» возвращает дефолт. Запомненный период «Сегодня» остаётся живым «сегодня».
- Тайл: фото (превью), подпись «№АЗС · категория · время», флажок «Отметить» (зона ≥40px, stopPropagation), бейдж «✓ замечание» (тап → мини-карточка: когда → кому → текст → кто отправил).
- Пагинация: страницы по 40, keyset-cursor, автоподгрузка по сентинелу + «Показать ещё». Превью грузятся лениво (только видимые, IntersectionObserver), blob-кэш с revoke.
- Категории = типы фото из CRM (как в requiredPhotos): код = id записи, подпись = title; если title не найден — показать код, не падать.

### 4.2 Полноэкранный просмотр (лайтбокс)
- Открытие тапом по тайлу. Свайпы (порог ~50px по X) и стрелки ←/→ листают в текущем порядке отфильтрованной ленты; Esc/крестик — закрыть; счётчик «3 / 47».
- Подпись: АЗС · категория · дата/время (EXIF при наличии). Кнопка «Отметить»/«Отмечено». Инфо-строка отправленного замечания, если есть.
- Достигнут конец загруженной части — догружается следующая страница. Предзагрузка соседних фото (n±1), revoke вне окна n±3. Скролл фона заблокирован.

### 4.3 Панель-черновик отправки
- Появляется при первой отметке (в сетке — sticky снизу, в лайтбоксе — нижний блок), НЕ блокирует листание.
- Содержимое: заголовок «АЗС №3 Северная · N фото», выпадашка «Кому»: **Управляющему (по умолчанию)** / Администратору АЗС — имена подставляются; кнопки-шаблоны (из настроек, дефолт — два из ТЗ) → подставляют текст в поле (редактируемый); textarea; «Отправить (N)»; «Очистить».
- Повторный тап «Отметить» снимает фото с черновика. Отправка: подтверждений не требуется (это один получатель одной АЗС); кнопка блокируется на время запроса (withPending-паттерн).
- После успеха: панель закрывается, выделение очищено, тост «Отправлено: Иванов П. (№3) — N фото», бейджи на фото.

### 4.4 Журнал (вкладка)
- Виды: «Лента» (обратная хронология) / «По АЗС» (группы с количеством, раскрываются). Фильтры: период, АЗС.
- Запись: дата-время · АЗС · получатель (имя, роль) · текст · миниатюры фото (тап → открыть фото в ленте/лайтбоксе) · кто отправил · статус доставки.
- Статус «не доставлено» — красная пометка + кнопка «Повторить» (повторная отправка той же записи).
- Журнал общий: видят все проверяющие и админы. Подпись в шапке: «Видно всем проверяющим и админам».

### 4.5 Настройки (новый блок «Фотолента»)
- Маппинг поля «Управляющий»: код UF-поля карточки АЗС (паттерн как у поля набора фото photoSet).
- Быстрые шаблоны сообщений: редактируемый список строк (добавить/изменить/удалить), дефолт — два текста из ТЗ.

## 5. Данные и API

### 5.1 Таблица `photo_remark` (новая; PG и MySQL ветки, ensureSchema-стиль)
`id, created_at, azs_id, azs_title (snapshot), recipient_role ('manager'|'admin'), recipient_user_id, recipient_name (snapshot), message (text), sender_user_id, sender_name (snapshot), photos (json: [{reportId, photoCode}]), delivery_status ('sent'|'failed'), delivery_error (null)`. Индексы: created_at desc, azs_id.

### 5.2 Эндпоинты (все — verifyToken + роли reviewer/admin)
- `GET /api/reports/photos/feed?dateFrom&dateTo&azsId[]&photoCode[]&remarks=all|with|without&limit≤100&cursor` → `{ items: [{reportId, azsId, azsTitle, photoCode, exifAt, uploadedAt, remark: {dt, recipientName, message, senderName} | null}], nextCursor }`. Бейджи приходят сразу (LEFT JOIN по последней записи журнала на фото) — без N+1.
- `GET /api/reports/photos/categories` → `{ items: [{code, title}] }` (CRM photoType, кэш 10 мин).
- `POST /api/photo-remarks` → `{ azsId, recipientRole, message, photos: [{reportId, photoCode}] }` → отправка через бота + запись журнала; ответ — созданная запись со статусом.
- `POST /api/photo-remarks/:id/retry` — повторная доставка записи со статусом failed.
- `GET /api/photo-remarks?view=time|azs&dateFrom&dateTo&azsId[]&limit&cursor` → записи журнала (+ для view=azs — группировка на фронте по полю azs_id; бэк отдаёт плоско).
- `GET /api/reports/photos/:reportId/:photoCode/preview` — существующий прокси превью, без изменений.

### 5.3 Резолв получателя
- Управляющий: чтение карточки АЗС из CRM, поле из настроек (`settings.azs.fields.manager`), значение — пользователь Б24. Кэш карточек на запрос.
- Администратор АЗС: существующая привязка (как у рассылки заданий).
- Резолв происходит при открытии панели (фронт запрашивает имена через feed/отдельный лёгкий вызов в составе POST — имена в панели берём из ответа `GET /api/reports/photos/recipients?azsId=` → `{manager: {id, name} | null, admin: {id, name} | null}`); пустой управляющий → С5.

## 6. Интеграция с Битрикс24 — отправка

**Отправитель — бот** (продуктовое решение: единообразно с заданиями и просрочками; технически бот гарантированно пишет в личку любому сотруднику).

- Основной путь: `imbot.v2.File.upload` — по одному вызову на файл (`fields.FILE = {name, content(base64 ≤100МБ), message}`); текст замечания («Замечание по АЗС №3 (Сергеев В.): {текст}») прикрепляется к первому файлу, остальные без текста. Контент берётся с Диска существующим `downloadFileContent` (таймаут 120с уже настроен).
- Все вызовы — через существующий `callWithTransientRetry` (429/503/таймауты обработаны спринтом 1). Частичный сбой (часть файлов ушла) → запись остаётся `failed` с описанием, «Повторить» шлёт ВСЁ заново (допускаем дубль фото в чате при повторе — осознанно, простота важнее).
- Запасной режим `PHOTO_FORWARD_MODE=commit` (env): `im.disk.file.commit` существующих FILE_ID под admin-контекстом — БЕЗ перезаливки, но отправителем становится человек-админ, не бот. Включается только осознанно, если на смоуке bot-upload окажется неприемлемо медленным. (Примечание: это сознательная правка чернового плана спринта, где commit был основным, — продуктовое решение «отправитель = бот» главнее.)

## 7. Ошибки, пустые состояния, производительность

- Все паттерны спринта 2 обязательны: скелетоны по сетке, ошибки загрузки с «Повторить», тосты успеха/ошибки, in-flight блокировки кнопок.
- Пустая лента: «Нет фото за выбранный период» + иконка. Пустой журнал: «Замечаний пока нет».
- Сбой справочника категорий — лента работает без чипов категорий, с видимой деградацией и retry (паттерн S2-03).
- Производительность: лениво-загруженные превью, ≤40 на страницу, blob-revoke; журнал постранично.

## 8. Критерии приёмки (демо продакту)

1. С1–С5 проходят end-to-end на dev-портале (включая получение сообщения управляющим в Битриксе: текст + фото открываются).
2. Фильтры комбинируются; «Сегодня» по умолчанию; запоминание фильтров между заходами работает; «Сбросить» возвращает дефолт.
3. Свайпы на телефоне (390px) листают плавно; отметка из лайтбокса и из сетки синхронна.
4. Журнал: обе вьюхи, переход миниатюра → фото; запись «не доставлено» с работающим «Повторить» (проверяется выключением сети/ботом).
5. Бейдж/чип-фильтр замечаний согласованы с журналом.
6. 20 фото × 1 получатель уходят без QUERY_LIMIT_EXCEEDED (ретраи отрабатывают), время приемлемое.
7. Тесты бэка зелёные (существующие + новые на стор/роуты/сервис отправки с фейками), lint/build фронта чистые.

## 9. Вне скоупа (зафиксировано с продактом)

- Свободный выбор получателя / групповые чаты (осознанное сужение формулировки ТЗ «в групповой чат или конкретным получателям» до двух ролей АЗС; вернёмся, если заказчик попросит).
- Персистентные метки на фото как «журнал нарушений» (наша «пометка» = выделение для отправки; история живёт в журнале замечаний).
- Статус «исправлено» от управляющего — V2: кнопка «Принято в работу» в сообщении бота (фундамент журнала это уже позволяет).
- Серверные thumbnails (S3-08, stretch) — стартуем на существующем прокси с ленивой загрузкой.
