# Spec: пересылка ответов оператора + аналитика причин (FEAT-C, фаза 1)

**Дата:** 2026-06-04
**Статус:** дизайн согласован, реализация отложена (прод заморожен, в гит без команды не пушим).
**Бэклог:** FEAT-2026-06-04-C.

## 1. Контекст и цель

По обратной связи внедрения нужно: (1) пересылать ответы оператора АЗС ответственным в один общий чат и (2) собирать **аналитику причин**, почему отчёт не сдан вовремя.

Текущее состояние (факты по коду):
- Бот зарегистрирован с `eventMode: 'fetch'` (`botRegistryService.js:65`), **обработчиков входящих нет** → бот сейчас только отправляет.
- Отчёт = CRM-айтем (`report.entityTypeId`); локально живут `dispatch_log`, `report_photo`, `dispatch_plan`, `crm_sync_jobs`.
- **Прод стирает локальную БД при каждом редеплое** (один контейнер на Timeweb). Принцип уже принят в DR-фиче: **Bitrix = сейф, локальная БД = рабочий кэш** (`dispatchPlanMirror`).

## 2. Объём

**Фаза 1 (эта спека):** захват структурированной причины (кнопки + «Другое: текст») + пересылка в один общий чат + хранение durable + аналитика причин. Подход **C** (выбран): сначала через приложение, без входящих событий бота.

**Вне объёма (фаза 2):** бот-нативный приём входящих (мгновенный тап в чате, пересылка любого текста) и **обратная сторона** (ответственный → оператору). Зависит от здоровых BUG-A/BUG-G.

## 3. Архитектура и durability

Причина снимается в приложении и пишется **в карточку отчёта CRM** — это источник истины, переживающий редеплой. Локальная таблица — кэш для быстрой аналитики, восстанавливаемый из CRM.

```
Бот (задание + кнопка-ссылка) ──▶ оператор тапает
        │
        ▼
/reason/<reportId> (мини-вью)  ──POST──▶  reasonRoutes
        │
        ├──▶ CRM: crm.item.update reason UF   (СЕЙФ, под живым токеном оператора) ← durable сразу
        ├──▶ report_reason (локальный КЭШ, структурно: code+text) ← быстрые счётчики
        └──▶ forwardingService ──▶ общий чат (bot send, фон/webhook-контекст) ← best-effort
```

**Ключевое свойство:** запись причины в CRM идёт **под живым токеном оператора** (свежий access_token сессии, без refresh) → durable немедленно и **не зависит от битого фон-контекста (BUG-G)**. Пересылка в чат — отдельный best-effort слой.

## 4. Данные

### 4.1 Источник истины — CRM UF
- Новое **строковое** пользовательское поле на смарт-процессе «Отчёт» (создаёт заказчик). Его код → настройка `report.fields.reason`.
- Кодировка значения (человекочитаемо, но обратимо):
  - пресет → его label, напр. `Очередь / много гостей`;
  - «Другое» → `Другое: <текст оператора>`.
- Запись через тот же путь, что и существующий sync полей отчёта (учесть camelCase↔`UF_CRM` алиасинг, как в `getFieldValue`).

### 4.2 Локальный кэш — таблица `report_reason` (PG + MySQL, dual-driver)
```
id, report_id (→dispatch_log.id), azs_id, admin_user_id,
reason_code, reason_text (nullable), source ('task'|'expiry'|'app'),
created_at, updated_at, UNIQUE(report_id)
```
upsert по `report_id` (последняя причина перезаписывает). Назначение — быстрые счётчики; не источник истины.

### 4.3 Rehydrate (после редеплоя)
`rehydrateReasonsIfEmpty`: если кэш пуст — прочитать у CRM-айтемов отчётов значение reason UF и восстановить кэш, обратно маппя label→code по каталогу; `Другое: ...` → code=`other`, text=остаток; нераспознанное → `other` + полный текст. По образцу `dispatchPlanMirror.rehydrateIfEmpty`.

### 4.4 Каталог причин — настройка `report.reasons[]` (БЕЗ хардкода)
**Принцип «никакого хардкода»:** ни код UF причины, ни список причин не зашиваются в бизнес-логику — источник только настройки (`app_settings` / `app.option`). В коде допустим лишь *seed* дефолта для первичного заполнения экрана настроек; вся логика (валидация, кнопки, маппинг, аналитика) читает исключительно `report.reasons[]` и `report.fields.*`. Список причин редактируется в экране настроек: добавить / убрать / переименовать / задать порядок. Дефолтный seed (заказчик меняет в UI):

| code | label |
|---|---|
| `fuel_truck` | Приёмка топлива / бензовоз |
| `delivery` | Приёмка товара |
| `queue` | Очередь / много гостей |
| `wc_busy` | Санузел занят |
| `staff` | Нехватка персонала |
| `other` | Другое (требует текст) |

### 4.5 Прочие настройки
- `report.fields.reason` — код строкового UF причины.
- `report.responsibleChatId` — id общего чата ответственных (+ опц. название). Пусто → пересылка пропускается, причина всё равно пишется.

## 6. Поток данных и API

**Захват:** `POST /api/reports/:reportId/reason` body `{ reasonCode, reasonText? }`
1. Доступ: оператор данного отчёта (`admin_user_id`) ИЛИ проверяющий.
2. Валидация: `reasonCode` ∈ каталог; при `other` — `reasonText` обязателен и непустой.
3. Запись CRM UF (под контекстом запроса = живой токен оператора).
4. upsert в `report_reason` (структурно).
5. best-effort `forwardingService` → сообщение в `responsibleChatId`:
   `«АЗС {name}: {operator} — причина: {label}{: text}. Отчёт {status}, дедлайн был {time}.»` + ссылка на карточку.
6. Ответ `{ ok: true }` даже если шаг 5 не удался (причина сохранена).

**Аналитика:** `GET /api/reports/reasons?dateFrom&dateTo&azsIds` → счётчики по `reason_code` (+ доля). Из кэша; при пустом кэше — `rehydrateReasonsIfEmpty`. Стыкуется с FEAT-E (разрез «Причины просрочек», причина рядом с `expired` в ленте).

**Триггеры:**
- **В задании:** клавиатура `notifyDispatch` получает кнопку-ссылку на `/reason/<reportId>` (проактивно).
- **При просрочке:** `timeoutWatcher` после `expired` проверяет наличие причины (кэш/CRM); если нет — бот шлёт оператору сообщение-добор с той же ссылкой.

## 7. Зависимости и порядок включения

- **Работает сразу (после реализации + разморозки):** захват причины + запись в CRM (живой токен оператора) + локальная аналитика. Durable независимо от BUG-G — **при условии, что у оператора есть права на правку своего отчёта** (ожидаемо, он ответственный); иначе durability обеспечивается ретраем записи под фон/webhook-контекстом (см. §8), и тогда зависит от фикса BUG-G.
- **«Загорается» после фикса BUG-G:** пересылка в чат (bot send требует здорового фон/webhook-контекста). До этого — причина пишется и видна в отчётах, пересылка best-effort падает в лог.
- **От заказчика:** строковый UF на отчёте + его код в `report.fields.reason`; id чата в `report.responsibleChatId`.

## 8. Ошибки и деградация

| Ситуация | Поведение |
|---|---|
| `report.fields.reason` не задан | писать только в кэш + warn (нет durability — настоятельно задать поле) |
| `responsibleChatId` пуст | пропустить пересылку, причина сохранена |
| CRM update упал (права/контекст) | сохранить кэш + log; (опц.) поставить durable-ретрай по образцу `crmSync` |
| Пересылка упала | log, захват не блокируется |
| Невалидный код / пустой текст при `other` | 400 с понятным сообщением |
| Повторный тап | upsert перезаписывает причину (и CRM, и кэш) |

## 9. Тестирование

- `reasonStore`: upsert, get-by-report, countsByCode за период, dual-driver (PG+MySQL).
- `reasonCatalog`: code→label и label→code, обработка `Другое: ...`, нераспознанное → other.
- route `POST /reason`: валидация (плохой код, пустой текст для other, доступ), запись CRM + кэш, триггер пересылки.
- route `GET /reasons`: счётчики; rehydrate при пустом кэше.
- `forwardingService`: компоновка сообщения, отправка в chatId, best-effort при сбое.
- `timeoutWatcher`: expired без причины → добор; с причиной → без добора.
- клавиатура задания: присутствует кнопка-ссылка.
- rehydrate: восстановление кэша из CRM-значений (round-trip кодировки).

## 10. Фаза 2 (эскиз, НЕ сейчас)

Бот-нативный приём: перерегистрация бота на `ONIMBOTMESSAGEADD` (push) + endpoint `/bitrix/bot-event` с проверкой подписи → мгновенный тап причины в чате и пересылка любого текста оператора. Обратная сторона (ответственный → оператору) через маппинг `отчёт ↔ тред чата ↔ оператор`. Хранилище/пересылка/аналитика из фазы 1 переиспользуются. Зависит от здоровых BUG-A/BUG-G. Требует отдельного брейншторма по обратной маршрутизации.
