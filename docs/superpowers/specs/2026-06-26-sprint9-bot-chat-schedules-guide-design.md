# Спринт 9 — Только бот в чат, встройка в чат, управление расписаниями, юзер-гайд

Дата: 2026-06-26
Ветка: `feature/sprint9-bot-chat-schedules-guide` (от `origin/master`)
Стек прод: Node API (`backends/node/api`) + Nuxt-фронт (`frontend`) + Postgres

## Контекст и цель

Четыре независимых рабочих элемента одного фикс-спринта. Каждый самодостаточен,
их можно делать и выкатывать по отдельности. Прод = master (FF-пуш + bump
`BUILD_REV` обе стадии, см. [[azs-git-deploy-workflow]]).

Роли в системе ([`roleResolver.js`](../../../backends/node/api/src/access/roleResolver.js)):
- `admin` — портал-админ (capabilities: settings + reviewer + reports);
- `reviewer` — управляющий (reviewer);
- `azs_admin` — администратор АЗС / оператор (reports).

## РЭ-1. Все уведомления — только сообщением бота в чат (колокольчик вырезаем)

### Проблема (зачем)
Пользователи **не слышат** колокольчик Битрикс (`im.notify.personal.add`), но
**всегда слышат** сообщение в мессенджере. Значит единственный канал доставки всех
сообщений приложения (пора сдать отчёт; срок профукан; и любые иные) — **сообщение
бота в чат**. Колокольчик недопустим даже как фоллбэк.

### Текущее состояние
Файл [`notificationService.js`](../../../backends/node/api/src/notifications/notificationService.js):
- режим по умолчанию `mode = process.env.BITRIX_BOT_MODE || 'notify'` (стр. 85) — колокольчик;
- `fallbackToNotify = true` (стр. 122) — при любом сбое бота вызывается
  `sendViaNotify` → `im.notify.personal.add` (стр. 72–81, 224, 234) → колокольчик;
- `im.notify.personal.add` физически вызывается в
  [`bitrixRestClient.js`](../../../backends/node/api/src/dispatch/bitrixRestClient.js) `notifyUser()` (~стр. 506–518).

Прямые/косвенные вызовы фоллбэка: `dispatchService.js:288`, `timeoutWatcher.js:124–169`,
`reportsRoutes.js:1726`.

### Решение
1. Режим по умолчанию → `bot` (стр. 85).
2. **Удалить путь доставки конечным сотрудникам через `sendViaNotify`/`notifyUser`.**
   `im.notify.personal.add` не вызывается ни для одной рассылки сотрудникам.
3. Сбой доставки бота (auth/token/`BOT_TOKEN_NOT_SPECIFIED`/OAuth) → вместо колокольчика:
   - **бот пишет в сервис-чат портал-админам** — DM ботом каждому из
     `access.adminUserIds` (фоллбэк env `SYSTEM_ADMIN_USER_IDS`/`ADMIN_USER_IDS`):
     «Не удалось доставить сообщение сотруднику {userId} по АЗС {azsId}. Причина: {botError}».
   - Алерт админу — **тоже сообщение бота** (тот же `sendViaBot`), не колокол.
4. Если и бот-алерт не ушёл (бот совсем недоступен) → `logger.error('notification_undelivered', …)`
   + существующий `bot_delivery_auth_problem`-лог. Тихих потерь нет — сбой виден в логе и (если возможно) в чате админа.
5. Параметр `fallbackToNotify` и связанный `fallbackSuffix` удаляются как мёртвые
   (колокольчик-фоллбэка больше нет, кнопки бота не теряются → суффикс «/reason N» не нужен).

### Что НЕ делаем (YAGNI)
- Не чиним здесь корневую OAuth-протухаемость (BUG-022) — это отдельная задача;
  bot-only делает сбой **видимым** (алерт), что и нужно.
- Не вводим очередь ретраев (пользователь выбрал «алерт админу», не «ретрай»).

### Единицы и интерфейсы
- `notificationService.notify()` — единственная точка выбора транспорта; после правок
  ветка одна: бот → (при сбое) бот-алерт админам → (при полном сбое) лог.
- Новая внутренняя функция `alertAdmins({ bitrixClient, botId, adminUserIds, text, context })`
  в notificationService — переиспользует `sendViaBot`.
- `adminUserIds` прокидывается в `createNotificationService` из настроек доступа
  (там же, где их читает `roleResolver`).

### Тестирование
- Юнит: режим по умолчанию = bot; успешная доставка → channel `bot`;
- сбой бота → `alertAdmins` вызван, `im.notify.personal.add` НЕ вызван;
- сбой бота И сбой алерта → `notification_undelivered` залогирован, исключений наружу нет;
- grep-гард в тесте/проверке: `notifyUser`/`im.notify` не вызывается на пути рассылки.

## РЭ-2. Встройка `IMMOBILE_CONTEXT_MENU` → всегда дашборд управляющего

### Цель
Пункт в контекстном меню мобильного чата Битрикс, открывающий интерфейс
администратора АЗС (дашборд управляющего, `/reviewer`).

### Текущее состояние
- Бэк [`server.js`](../../../backends/node/api/server.js): `ensureRestAppUriPlacement()` (~141–215)
  привязывает только `REST_APP_URI` в install-флоу (~838–859). Других встроек нет.
- Фронт [`index.client.vue`](../../../frontend/app/pages/index.client.vue):
  `parsePlacementOptions()` (199–215), роутинг по контексту (336–382). `PLACEMENT`/
  `PLACEMENT_OPTIONS` сейчас читаются только при install и не используются для навигации.

### Решение
1. **Бэк:** обобщить привязку встроек — рядом с `REST_APP_URI` биндить
   `IMMOBILE_CONTEXT_MENU` (`placement.bind`, handler = тот же `APP_BASE_URL`/`VIRTUAL_HOST`,
   заголовок «Порядок на АЗС»). Идемпотентно (как уже сделано для `REST_APP_URI` через `placement.get`).
   На реинсталле/переустановке — биндить обе.
2. **Фронт:** в роутинге `index.client.vue` определить `placement.title === 'IMMOBILE_CONTEXT_MENU'`
   → навигация на `/reviewer`.
3. **Ролевое гейтирование сохраняется:** если открыл пользователь без права `reviewer`
   (оператор) — показываем понятное «Этот раздел для управляющего», а не пустой экран.

### Открытый технический момент
Точный набор опций, которые Битрикс передаёт в `IMMOBILE_CONTEXT_MENU` (DIALOG_ID и т.п.),
уточняется при реализации эмпирически/по докам; для решения «всегда дашборд» они не
обязательны — роутим на `/reviewer` без зависимости от опций.

### Тестирование
- Install-флоу биндит обе встройки (мок `placement.bind`, проверка вызова с `IMMOBILE_CONTEXT_MENU`);
- повторный install не дублирует (placement.get → already exists);
- фронт: при `PLACEMENT=IMMOBILE_CONTEXT_MENU` роут = `/reviewer`;
- оператор без reviewer → экран-заглушка, не краш.

## РЭ-3. Управление расписаниями: план на день + глобальное расписание

### Цель
Удобно создавать/отменять/пересоздавать запросы отчётов. Бэк уже умеет почти всё,
UI не показывает.

### Текущее состояние
- Хранилища: `dispatch_log` ([`dispatchLogStore.js`](../../../backends/node/api/src/dispatch/dispatchLogStore.js)),
  `dispatch_plan` ([`dispatchPlanStore.js`](../../../backends/node/api/src/reports/dispatchPlanStore.js)).
- Настройки: `report.dispatchTimes[]`, `dispatchJitterMinutes`, `timeoutMinutes`
  ([`defaultSettings.js`](../../../backends/node/api/src/settings/defaultSettings.js)).
- Эндпоинты есть: `POST /api/reports/manual` (ручной запрос), `POST /api/reports/today/reissue`
  (перевыпуск дня), `POST /api/reports/plan/generate`. Чтения плана дня и точечной отмены/
  пересоздания слота в REST нет.
- UI: карточка «Расписание рассылки» в [`reviewer.client.vue`](../../../frontend/app/pages/reviewer.client.vue) (~1257–1347) —
  плоский список времён, сохранение через `PUT /api/settings`. Нет списка запросов, нет отмены/пересоздания.

### Решение
**Бэк — добавить REST-обёртки (логика в сторах уже есть):**
- `GET /api/reports/plan?date=YYYY-MM-DD` → список слотов дня из `dispatch_plan`/`dispatch_log`:
  `{ azsId, azsTitle, time, status, slotKey, planId }`.
- `POST /api/reports/plan/slot/cancel` `{ planId | slotKey, date }` → отменить отдельный
  запланированный слот (статус → `cancelled`, не отправлять).
- `POST /api/reports/plan/slot/reissue` `{ azsId, date, time }` → пересоздать/поставить
  заново отдельный слот.
- Доступ: reviewer ИЛИ settings (как у dispatch-роутов).

**Фронт — переработать блок расписания в карточке управляющего:**
- Подблок «Глобальное расписание»: понятный редактор времён + джиттер + таймаут (как есть, аккуратнее).
- Подблок «План на день»: выбор даты → таблица слотов (АЗС / время / статус) с действиями:
  - отменить слот;
  - пересоздать слот;
  - «Перевыпустить на сегодня» (на существующий `today/reissue`);
  - «Создать вне расписания» (на существующий `manual`).

### Единицы и интерфейсы
- Новый файл фронта-компонента `ScheduleManager.vue` (или секция в reviewer) — изолировать
  логику плана дня от остальной карточки управляющего, т.к. reviewer.client.vue уже большой.
- Стора `api.ts` — методы `getDayPlan`, `cancelSlot`, `reissueSlot`.

### Что НЕ делаем (YAGNI)
- Профили расписаний по каждой АЗС — отдельный крупный слой, в этот спринт не входит.

### Тестирование
- Бэк: `GET /api/reports/plan` возвращает слоты дня; cancel переводит слот в `cancelled`
  и он не отправляется тиком планировщика; reissue ставит слот заново;
- доступ: оператор без reviewer/settings → 403;
- фронт: таблица рендерит слоты, действия дёргают нужные эндпоинты, оптимистично обновляют статус.

## РЭ-4. Юзер-гайд — переписать и расширить существующий

### Цель
Подробный актуальный гайд, отдельный раздел на каждую роль, доступный со всех экранов.

### Текущее состояние
- Уже есть: [`HelpGuide.vue`](../../../frontend/app/components/help/HelpGuide.vue) (3 вкладки:
  Администратор АЗС / Управляющий / Настройки), [`HelpButton.vue`](../../../frontend/app/components/help/HelpButton.vue),
  [`useHelpDrawer.ts`](../../../frontend/app/composables/useHelpDrawer.ts), мокапы.
- Кнопка «Справка» сейчас только на reviewer ([`reviewer.client.vue`](../../../frontend/app/pages/reviewer.client.vue) ~881).
- Контент частично устарел (нет фотоленты, брендов, нового управления расписаниями, встройки в чат, «только бот»).

### Решение
1. Переписать контент трёх вкладок под текущие фичи; для каждой роли — подробно:
   зачем, как приходит запрос (через бота в чат!), как сдать отчёт, статусы, что настраивать.
   Добавить разделы: фотолента, бренды/внешний доступ, управление расписаниями (план дня),
   встройка в чат, «уведомления только через бота».
2. Вывести `HelpButton` на **все** экраны: `index`, `settings`, `reviewer`, `reports`,
   `brands`, `admin/[reportId]` — с `default-role` под экран.
3. Сохранить структуру вкладок (3 роли), мокапы обновить при необходимости.

### Что НЕ делаем (YAGNI)
- Новый полноэкранный раздел-гайд с нуля — переиспользуем существующий drawer.

### Тестирование
- Кнопка «Справка» видна и открывает drawer на всех 6 экранах;
- открытие с правильной вкладкой под роль/экран;
- контент без пустых/устаревших разделов (само-ревью спека).

## Порядок реализации
РЭ-1 (уведомления — самый важный для пользователя) → РЭ-3 (расписания) →
РЭ-2 (встройка) → РЭ-4 (гайд, отражает уже сделанное). Каждый РЭ — отдельный коммит,
по готовности — FF в master + bump `BUILD_REV` обеих стадий, проверка смены имён `_nuxt`-чанков.

## Пост-деплой (из памяти)
JOB_SECRET в env + «Перерегистрировать бота»; для bot-only критично, чтобы бот был
зарегистрирован и `BITRIX_BOT_MODE=bot` в проде.
