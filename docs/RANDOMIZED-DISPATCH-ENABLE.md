# Рандомизированный план рассылки — инструкция по включению

Фича «plan-then-execute»: каждая АЗС получает запрос отчёта в своё случайное
время в рамках рабочего окна. АЗС не знают заранее — внезапная проверка.
Каждый день — новый рандом. Проверяющий видит справочный «План на сегодня».

**По умолчанию ВЫКЛЮЧЕНА** (`DISPATCH_PLAN_MODE_ENABLED=false`). Пока флаг off —
работает старое поведение, прод не меняется.

---

## 1. Перед включением

- `.env`: `SCHEDULER_ENABLED=true` (плановый рассыльщик должен быть включён).
- В Настройках приложения (экран настроек в Битрикс24) задать:
  - **Рабочее окно** рассылки, напр. `07:00`–`22:00` — раньше начала и позже
    конца запросы не уходят.
  - **Времена авто-отправки** (base-времена), напр. `09:00`.
  - **Разброс (джиттер)**, минут, напр. `240` = ±4 часа от base-времени.

## 2. Включение

В `.env`:
```
DISPATCH_PLAN_MODE_ENABLED=true
```
Опционально:
```
DISPATCH_PLAN_GENERATION_CRON='1 0 * * *'   # когда генерится план (00:01)
DISPATCH_EXECUTE_BATCH_LIMIT=20             # макс. отправок за одну минуту
```
Затем **рестарт приложения**. На старте:
- создастся таблица `dispatch_plan`;
- сгенерируется план на остаток текущего дня (идемпотентно — повторный
  рестарт не задвоит).

## 3. Проверка

1. Экран проверяющего → карточка **«План отчётов на сегодня»**: список
   АЗС | время запроса (МСК) | ответственный | статус. У каждой АЗС своё
   случайное время.
2. Логи: `dispatchScheduler: plan generation cron started`, далее
   `executeDuePlans` по мере наступления времён, `markDispatched` после отправки.
3. Дождаться первого времени → у соответствующей АЗС должен прийти запрос
   (бот-уведомление + элемент в смарт-процессе отчётов).

## 4. Откат (мгновенный)

```
DISPATCH_PLAN_MODE_ENABLED=false
```
→ рестарт → сразу старое поведение (рассылка всем в минуту base-времени).
Таблица `dispatch_plan` остаётся, но не используется.

---

## Важно

- **Время — московское** (берётся из `settings.timezone`). `09:00` означает
  09:00 МСК, как и в старом режиме (баг с UTC-сдвигом исправлен до релиза).
- **Перед боевым включением** рекомендуется e2e-смоук на стейдже: включить флаг,
  убедиться что план генерится и АЗС получают запросы в свои случайные времена.
- Безопасность: при флаге off код рассылки не меняется (проверено побайтово),
  таблица плана не создаётся. Риск для прода при выключенном флаге — нулевой.
